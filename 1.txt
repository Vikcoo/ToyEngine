# SceneProxy 设计问题整理

## 1. SceneProxy 中混入了 GPU 资源创建逻辑

### 问题描述

当前 `FStaticMeshSceneProxy` 在构造过程中直接创建 Shader、Pipeline、VertexBuffer 和 IndexBuffer。这使得 SceneProxy 不再只是“渲染代理对象”，而同时承担了 GPU 资源初始化职责。

### 原因

SceneProxy 的职责边界没有和 RenderResource、Renderer 管理层分开，导致“场景代理层”和“底层资源层”被写进了同一个对象中。

### 推荐改进方式

将 GPU 资源创建逻辑从 SceneProxy 中拆出。SceneProxy 只负责描述这个场景物体应如何参与渲染，例如持有 mesh、material、transform、bounds、render flags 等信息；而 GPU Buffer 等资源应交给独立的 RenderResource 层负责。

---

## 2. SceneProxy 直接依赖 RHIDevice，分层不合理

### 问题描述

当前 SceneProxy 构造函数直接接收 `RHIDevice*`，说明 SceneProxy 需要了解底层设备对象，并直接通过它创建 GPU 资源。

### 原因

Scene 层、Renderer 层、RHI 层之间的边界没有清晰拆开，导致 SceneProxy 直接与底层设备层耦合。

### 推荐改进方式

不要让 SceneProxy 直接依赖 `RHIDevice`。更合理的方式是让它依赖已经准备好的渲染资源描述，或者依赖更高层的渲染资源对象。这样可以让 SceneProxy 保持轻量，并避免 scene 层直接绑定到底层 RHI 实现。

---

## 3. Shader / Pipeline 的创建位置不合理

### 问题描述

当前 SceneProxy 在构造时直接创建 Shader 和 Pipeline，这意味着每个 SceneProxy 都可能单独承担一份渲染状态创建工作。

### 原因

Shader、Pipeline 这类资源被当成了“场景对象私有资源”，而不是“渲染系统统一管理和复用的资源”。

### 推荐改进方式

将 Shader 和 Pipeline 的创建与缓存从 SceneProxy 中移除，交由 Renderer 或 Material System 统一管理。SceneProxy 只保留与材质、渲染配置相关的轻量描述，而不直接承担 Pipeline 的生命周期管理。

---

## 4. Mesh GPU 资源放在 SceneProxy 中，不利于实例复用

### 问题描述

当前实现中，每个 SceneProxy 都会根据 StaticMesh 的 section 数据创建自己的 VBO 和 IBO。这意味着相同的 StaticMesh 被多个场景实例引用时，可能会重复上传多份完全相同的 GPU 数据。

### 原因

网格资源被当成了“实例级数据”，而不是“资产级共享资源”，导致几何数据无法在多个实例之间复用。

### 推荐改进方式

将 Mesh 的 GPU 资源提升到 StaticMesh 对应的共享 RenderResource 中，由 StaticMesh 的多个实例共同引用。SceneProxy 只保存实例级信息，例如变换、材质覆盖、可见性状态等，而不是自己拥有一整套几何 GPU 资源。

---

## 5. 每个 Section 单独创建 VBO/IBO，容易造成资源碎片化

### 问题描述

当前为每个 section 分别创建顶点缓冲和索引缓冲，这会导致一个 mesh 拆分出多个小型 GPU 资源对象。

### 原因

section 被直接映射为独立资源，而不是作为同一份 mesh GPU 数据中的逻辑子区间来管理。

### 推荐改进方式

更合理的方式是让整个 mesh 使用更集中、更统一的 GPU 资源表达，再由 section 记录各自的范围信息。这样能够减少资源碎片化，也更利于后续做资源管理、批量上传和渲染提交。

---

## 6. 构造函数中通过 return 提前退出，会留下半初始化对象

### 问题描述

当前构造函数在检测到设备无效、mesh 无效、pipeline 创建失败等情况时直接 `return`。在 C++ 中，这不会让对象真正“构造失败”，而是留下一个已经生成但状态不完整的对象。

### 原因

对象创建失败流程没有和对象构造流程严格分离，导致错误状态只能靠对象内部的“是否有效”标记来兜底。

### 推荐改进方式

避免在构造函数中使用“提前 return + 半初始化对象”的方式表示失败。应将失败路径设计得更明确，例如通过更清晰的创建流程或初始化流程来保证无效对象不会被错误地注册进渲染系统。

---

## 7. Buffer 创建缺少完整的失败检查

### 问题描述

当前创建 VertexBuffer 和 IndexBuffer 后，没有清晰体现对失败情况的完整处理。如果部分资源创建成功、部分失败，就可能留下状态不完整的 section 数据。

### 原因

资源创建流程缺少细粒度的结果校验和失败回滚策略。

### 推荐改进方式

在资源创建过程中对每一步的结果进行明确检查，并在失败时有一致的处理策略。避免把“部分有效”的资源状态继续传递到后续渲染流程中。

---

## 8. IsValid 的语义容易变得模糊

### 问题描述

当前对象承担的职责较多，最终会导致 `IsValid()` 这种状态判断变得很难定义。例如，一部分资源成功、一部分资源失败时，到底算不算有效，很容易变得含糊。

### 原因

一个对象承担了过多职责，导致“有效性”不再对应单一标准，而是被多个不同层面的状态混在一起判断。

### 推荐改进方式

通过职责拆分降低状态判断的复杂度。让资源层、代理层、渲染状态层各自管理自己的有效性，这样每个对象的状态语义都会更清晰，也更容易排查问题。

---

## 9. 资源释放责任不明确

### 问题描述

当前 SceneProxy 创建了 Buffer、Shader、Pipeline 等多类资源，但从设计上看，这些资源究竟由谁拥有、由谁释放，并不清晰。

### 原因

资源所有权和对象职责边界没有划清，导致 SceneProxy 同时像是“渲染代理对象”又像是“资源拥有者”。

### 推荐改进方式

明确资源所有权。Mesh GPU 资源应由 RenderResource 层管理，Shader / Pipeline 应由 Renderer 或 Material System 管理，SceneProxy 只持有必要的引用或描述信息，而不承担所有资源的归属与释放责任。

---

## 总结

当前实现作为原型是能工作的，但如果目标是向 UE5 风格的引擎架构靠拢，核心问题在于：

SceneProxy 被设计成了一个同时负责“场景代理、资源上传、渲染状态创建”的重对象。

这会导致：

* Scene 与 RHI 耦合过深
* Mesh 资源无法高效共享
* Shader / Pipeline 生命周期不清晰
* 对象状态和资源归属难以维护
* 后续扩展渲染线程、实例化渲染、资源流送时重构成本很高

更合理的方向是：

* SceneProxy 只负责描述“如何渲染”
* RenderResource 负责管理“GPU 上有哪些资源”
* Renderer / Material System 负责“使用什么 Shader / Pipeline”

这样整体结构会更清晰，也更接近 UE5 的设计思路。
