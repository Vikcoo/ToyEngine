# 运行时模块

本文档说明当前实际参与构建的运行时模块，以及它们在项目中的职责边界。

## 当前构建顺序

`Source/Runtime/CMakeLists.txt` 当前按以下顺序加入模块：
- `Core`
- `Platform`
- `Input`
- 启用 `TE_RHI_OPENGL` 时加入 `RHIOpenGL`
- `RHI`
- `Asset`
- `RenderCore`
- `World`
- `Renderer`
- `Engine`

这里需要注意：`RHI` 当前仅包含抽象接口，不再反向依赖具体后端。具体后端的创建逻辑由更外层装配模块处理（当前放在 `Engine` 中）。

## 模块说明

### Core
路径：`Source/Runtime/Core`

职责：
- 数学类型与变换
- 日志系统
- 内存辅助能力
- 通用底层工具

当前接口边界：
- `Core/Public/Math` 对外暴露的是引擎自有数学类型（`Vector*`、`Matrix*`、`Quat`）
- `glm` 仅作为 `Core` 私有实现细节存在于 `Private` 中，不应出现在其他运行时模块的公开接口里
- `Core/Public/Math/ScalarMath.h` 提供轻量标量数学工具，不依赖向量/矩阵类型
- `Core/Public/Math/MathUtils.h` 当前保留为向量扩展数学工具与兼容入口，依赖 `MathTypes.h`
- `Core/Public/Log` 对外暴露的是引擎自有日志接口与日志宏
- `spdlog` 仅作为 `Core` 私有实现细节存在于 `Private` 中，不应出现在其他运行时模块的公开接口里
- 日志同时写入彩色控制台和引擎根目录下的 `Saved/Logs/ToyEngine.log`；文件达到 `5 MiB` 后滚动，最多保留 3 个历史文件
- `Log::Shutdown()` 采用有序停机契约：所有可能写日志的线程停止后，由应用层在最后一条日志之后调用，确保异步队列刷新并回收后台线程
- `Core/Public/Memory/Memory.h` 暴露默认内存系统入口；`MemoryShutdown()` 采用有序停机契约，调用前必须停止所有可能使用 `MemAlloc` / `MemFree` 的 worker，并释放旧裸指针
- `Core/Public/Memory/MemoryUtils.h` 当前仅暴露内存工具声明；日志输出实现位于 `Private/Memory/MemoryUtils.cpp`
- 依赖日志能力的代码应显式包含 `Core/Public/Log/Log.h`，不要依赖 `MemoryUtils.h` 的间接包含

#### 日志持久化实现

日志初始化的数据流为 `TE_LOG_* -> Log -> spdlog::async_logger -> console sink + rotating file sink`。`Core` 的 CMake 目标通过私有的 `TE_ENGINE_ROOT_DIR` 编译定义提供引擎根目录，日志实现据此构造稳定的绝对路径，不依赖 IDE 或命令行启动时的工作目录。`Saved/Logs` 由初始化流程自动创建，并在 `.gitignore` 中整体排除。

文件输出采用滚动策略，而不是无限追加的基础文件 sink：当前文件上限为 `5 MiB`，保留 3 个历史文件，在保留最近排查信息的同时限制磁盘占用。日志仍使用容量为 8192 的阻塞式异步队列，错误及以上级别会触发刷新，普通日志每 3 秒定期刷新；正常退出时仍必须调用 `Log::Shutdown()`，以覆盖定时刷新间隔内尚未落盘的内容。

当前方案适用于从源码树运行的开发阶段。它的限制是日志目录由配置时的源码根目录决定；未来制作可搬运或只读安装包时，应改由运行时路径服务选择用户可写目录，而不是继续依赖构建期路径。后续排查优先查看 `Source/Runtime/Core/Private/Log.cpp`、`Source/Runtime/Core/CMakeLists.txt` 与本文档。

### Platform
路径：`Source/Runtime/Platform`

职责：
- 窗口抽象
- 事件轮询
- 缓冲交换与帧缓冲尺寸查询
- 通过 `IWindow::PresentCallback` 为 OpenGL Device 提供不泄漏 Platform 类型的呈现桥接
- 输入相关回调的注册入口

当前实现：
- `Private/GLFW` 下提供基于 GLFW 的窗口实现

### Input
路径：`Source/Runtime/Input`

职责：
- 键盘状态与过渡态管理
- 鼠标按键状态管理
- 鼠标移动增量
- 滚轮增量

核心类型：
- `InputManager`

### RHI
路径：`Source/Runtime/RHI`

职责：
- 提供与图形后端无关的渲染接口
- 抽象设备、命令缓冲、缓冲区、Shader、Pipeline
- 抽象 2D 纹理、Cubemap 纹理与采样器
- 提供公共渲染枚举与描述结构
- 定义 `BeginFrame/EndFrame/WaitIdle` 帧生命周期，以及当前帧命令缓冲和呈现状态
- 提供按帧分段的 transient uniform 分配接口、`DynamicUniformBuffer` 与 dynamic offset 绑定语义
- 提供显式资源 usage/memory、纹理状态转换、RenderPass load/store 和 Pipeline attachment/blend 描述
- `RHIShaderDesc` 只描述逻辑 Shader 名称、阶段与入口点，不暴露后端原生文件路径
- 提供 `RHIBindGroupLayout` / `RHIPipelineLayout`，将资源绑定接口形状显式化
- 提供 `RHIDevice::Create()` 声明，由外层模块实现具体后端装配逻辑

### RHIOpenGL
路径：`Source/Runtime/RHIOpenGL`

职责：
- 为 `RHI` 接口提供 OpenGL 实现
- 作为当前实际启用的图形后端
- 提供 OpenGL 后端装配入口（`CreateRHIOpenGLDevice()`），供外层模块调用
- 通过 `OpenGLShaderLibrary` 将 Renderer 请求的逻辑 Shader 名称解析为 OpenGL GLSL 资产
- 实现 `OpenGLBindGroupLayout` / `OpenGLPipelineLayout`，并在创建 BindGroup / Pipeline 时做轻量结构校验
- 支持 `GL_TEXTURE_2D` 与 `GL_TEXTURE_CUBE_MAP`，供材质贴图、GBuffer、天空盒和 IBL 使用
- 以单帧 CommandBuffer 实现 Device 帧协议，通过平台回调呈现，并使用 `glBindBufferRange` 实现动态 Uniform 范围
- 展开共享 GLSL 的受限相对路径 include；在 OpenGL 下把 group/set 宏降级为全局 binding

状态：
- 当前已实现，并在开启 OpenGL 选项时参与编译

### Asset
路径：`Source/Runtime/Asset`

职责：
- 管理 CPU 侧静态网格数据
- 通过 Assimp 导入模型文件
- 组织多 Section 网格数据
- 提取并保存最小 PBR 材质槽描述，包括 BaseColor / Metallic / Roughness / Normal / AO / Emissive 的因子与贴图路径

核心类型：
- `StaticMesh`
- `FMaterial`
- `FAssetImporter`

### RenderCore
路径：`Source/Runtime/RenderCore`

职责：
- 承载游戏侧和渲染器都需要引用的共享渲染类型
- 定义 `FPrimitiveComponentId`（跨游戏侧/渲染侧的稳定 Primitive 身份）
- 定义 `FLightComponentId`（跨游戏侧/渲染侧的稳定 Light 身份）
- 定义 `FPrimitiveSceneProxy`、`FStaticMeshSceneProxy`
- 定义 `FLightSceneProxy`（方向光与点光的渲染侧镜像）
- 定义 `FPrimitiveSceneInfo`（场景注册节点）
- 定义 `FStaticMeshRenderData` 与 `FMeshDrawCommand`（含结构化 `FPipelineKey`、材质槽索引与资产引用描述）
- 作为 `World` 与 `Renderer` 之间更稳定的公共渲染数据边界

### World
路径：`Source/Runtime/World`

职责：
- 管理游戏侧世界、Actor 与组件
- 维护组件变换
- 管理 Primitive 注册与脏状态
- 管理 Light 注册与脏状态
- 通过 `CreateSceneProxy` 直接声明具体渲染代理类型
- 通过 `CreateLightSceneProxy` 声明光源渲染代理
- 通过 `IRenderScene` 将游戏侧状态同步到渲染侧
- 不再通过 `IRenderScene` 直接请求静态网格 RenderData / Pipeline
- 提供相机组件与飞行相机控制器

核心类型：
- `IRenderScene`
- `World`
- `Actor`
- `Component`
- `SceneComponent`
- `PrimitiveComponent`
- `MeshComponent`
- `LightComponent`
- `DirectionalLightComponent`
- `PointLightComponent`
- `CameraComponent`
- `FlyCameraController`

### Renderer
路径：`Source/Runtime/Renderer`

职责：
- 管理渲染侧场景数据
- 通过 `FScene`（`IRenderScene` 实现）接收游戏侧的注册/更新/销毁请求
- 维护 Primitive 与 Light 两类渲染侧场景对象
- 在 Primitive 注册阶段触发资源准备
- 通过 `FRenderResourceManager` 管理 `FStaticMeshRenderData` 资产级 GPU 数据缓存、PBR 材质贴图缓存、运行时 IBL 环境资源与共享管线
- 通过 `RendererShaderNames` 请求逻辑 Shader，不查询后端类型或拼接后端 Shader 目录
- 通过 `RendererBindGroups` 与 `RendererBindings` 分离 BindGroup group index 和 shader binding slot
- 为 Forward BasePass、Deferred GBuffer 与 Deferred Lighting 创建显式 PipelineLayout
- `FSceneRenderer` 调度当前 `IRenderPath`，默认使用 `FForwardRenderPath`，并支持运行时切换到 `FDeferredRenderPath`
- `FForwardRenderPath` 通过 `FMeshPassProcessor(BasePass)` 构建绘制命令，并在提交阶段解析 `FPipelineKey -> Pipeline`，统一完成渲染状态绑定
- `FDeferredRenderPath` 执行 GBuffer Pass + Lighting Pass，并管理 GBuffer `RHIRenderTarget`
- Renderer 通过 `RendererDepthConvention` 将 CPU 正向 ZO 投影转换为 GPU Reversed-Z，并统一深度清除值 0 与 `Greater` 比较
- 在提交阶段通过纹理 `BindGroup` 绑定 BaseColor / Normal / Metallic / Roughness / AO / Emissive 材质贴图
- 在提交阶段通过 Environment `BindGroup` 绑定 irradiance cubemap、prefilter cubemap 与 BRDF LUT
- `FForwardRenderPath` 先绘制 Sky pass，再绘制静态网格 PBR
- 在提交阶段通过 transient Uniform ring 与 dynamic offset 绑定每个 draw 的 Object/Material 数据，并对 Light、DeferredPass 和 Sky 数据使用同一帧内快照协议
- Deferred GBuffer 在附件写入前后显式切换颜色/深度资源状态，再由 Lighting Pass 采样
- 收集并排序绘制命令
- 执行当前渲染路径的命令提交

核心类型：
- `FScene`
- `FRenderResourceManager`
- `FPrimitiveSceneInfo`
- `FViewInfo`
- `SceneRenderer`

### Engine
路径：`Source/Runtime/Engine`

职责：
- 持有并初始化各子系统
- 驱动主循环与单帧调度
- 将单帧调度拆为平台事件、输入、游戏侧更新、游戏到渲染同步、渲染阶段和帧尾收尾
- 通过 `RHIDevice::BeginFrame/EndFrame` 驱动后端帧生命周期，不持有全局 CommandBuffer，也不直接呈现
- 管理关闭顺序
- 提供应用层注入点（场景初始化回调、每帧更新回调、主相机注册）

核心类型：
- `Engine`

当前说明：
- `TickGameThread` / `TickRenderThread` 是单线程阶段命名，用于表达接近 UE5 的职责边界；当前尚未引入独立 GameThread / RenderThread。
