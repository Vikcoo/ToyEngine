# 架构优化复盘：渲染资源准备阶段解耦（第一阶段）

## 背景与问题

在本阶段开始前，`SceneProxy` 虽然已不直接 `CreateBuffer`，但静态网格 GPU 资源准备仍发生在 `TMeshComponent::CreateSceneProxy` 调用链里：
- 组件在创建代理时通过 `IRenderScene` 直接查询 `RenderData/Pipeline`
- `IRenderScene` 同时承担“场景同步接口”和“资源服务接口”
- `FScene` 同时承担场景容器、资源缓存、共享管线初始化多类职责

这会导致职责边界仍然偏混合：代理创建阶段在时序上仍夹带资源准备行为。

## 设计目标

- 让 `CreateSceneProxy` 回归“实例描述创建”语义
- 让资源准备进入独立阶段，并由渲染侧统一触发
- 收敛 `IRenderScene` 边界，只保留场景注册/更新/移除语义
- 将资源管理职责从 `FScene` 抽离到独立管理对象

## 核心方案与取舍

本阶段采用以下方案：
- `IRenderScene` 删除资源查询接口，只保留：
  - `AddPrimitive`
  - `UpdatePrimitiveTransform`
  - `RemovePrimitive`
- `TPrimitiveComponent::CreateSceneProxy` 改为无参数版本；`TMeshComponent::CreateSceneProxy` 仅创建携带 `TStaticMesh` 资产引用的代理
- 新增 `FRenderResourceManager`，集中管理：
  - `TStaticMesh -> FStaticMeshRenderData` 缓存
  - 静态网格共享 Shader/Pipeline
  - 代理资源准备（`PrepareStaticMeshProxy`）
- `FScene::AddPrimitive` 显式调用 `PrepareProxyResources`，资源就绪后才注册 `FPrimitiveSceneInfo`

取舍：
- 当前仍是单线程同步直调，不引入命令队列
- 只处理静态网格代理路径，未扩展到材质系统分层和多渲染路径资源策略

## 收益

- `SceneProxy` 创建阶段不再请求共享 GPU 资源，职责更贴近“实例描述 + 绘制接口”
- 资源准备阶段变得显式：`CreateSceneProxy` 与 `AddPrimitive` 两阶段边界更清晰
- `IRenderScene` 接口语义收敛，`World` 对 `Renderer` 的耦合进一步减小
- `FScene` 职责收敛到场景容器与注册流程，资源细节下沉到 `FRenderResourceManager`

## 当前限制与后续方向

- 仍是单线程同步调用，尚未引入 GT/RT 命令队列
- `FRenderResourceManager` 目前仍在 `Renderer` 模块内，后续可演进为更独立的资源子系统
- Pipeline 仍为静态网格共享通道，尚未按材质/渲染路径细分
- `FStaticMeshRenderData` 仍是“创建即上传”模型，后续可继续拆分 CPU 描述与 GPU 生命周期管理

## 增量记录：第二阶段（2026-04-12）

### 为什么继续改

第一阶段虽然把资源准备从 `CreateSceneProxy` 调用链中剥离出来，但 `FStaticMeshSceneProxy` 仍持有 `RHIPipeline*`，这会让实例镜像层继续携带渲染状态对象，不利于进一步贴近 UE5 的“描述在前、状态解析在后”思路。

### 第二阶段核心改动

- `FMeshDrawCommand` 从直接持有 `RHIPipeline*` 改为持有 `EMeshPipelineKey`。
- `FStaticMeshSceneProxy` 移除 `RHIPipeline*` 成员，只保留：
  - 静态网格资产引用
  - `FStaticMeshRenderData` 引用
  - 产出 `PipelineKey` 的绘制命令
- `FRenderResourceManager::PrepareStaticMeshProxy` 只注入 `RenderData`，并在注册阶段预热共享 Pipeline。
- `SceneRenderer` 在提交阶段通过 `FScene` 解析 `PipelineKey -> Pipeline`，再执行实际绑定与绘制。

### 收益

- SceneProxy 从“资源 + 状态句柄”的混合体进一步收敛为“实例描述 + 绘制描述输出”。
- DrawCommand 语义从“可直接提交的低层状态包”收敛为“渲染侧可解析的中间描述”。
- Pipeline 状态解析集中到 Renderer 侧，更接近 UE5 的 pass/processor 思路。

### 新限制

- 当前 `PipelineKey` 仍是单一静态网格通道，尚未接入材质维度的 key 组合。
- Pipeline 解析仍是单线程即时解析，尚未进入真正的 MeshPass 命令构建缓存体系。

## 增量记录：第三阶段（2026-04-12）

### 为什么继续改

前两阶段已经完成“Proxy 去资源 owner”和“PipelineKey 渲染侧解析”，但渲染输出仍只有顶点色路径，`MaterialIndex` 尚未驱动实际贴图绑定，无法绘制带纹理模型。

### 第三阶段核心改动

- RHI 抽象新增纹理能力：
  - 新增 `RHITexture` / `RHISampler` 接口
  - `RHIDevice` 增加 `CreateTexture` / `CreateSampler`
  - `RHICommandBuffer` 增加 `BindTexture2D` 与 `SetUniformInt`
- RHIOpenGL 后端新增：
  - `OpenGLTexture` / `OpenGLSampler` 实现
  - 命令缓冲支持纹理槽绑定与 `sampler2D` uniform 设置
- Asset 导入链路新增：
  - 导入 `aiMaterial` 的 BaseColor / Diffuse 贴图路径
  - `StaticMesh` 新增材质槽数据
- Renderer 资源管理新增：
  - `FRenderResourceManager` 缓存 `StaticMesh + MaterialIndex -> BaseColor Texture`
  - 提供默认白纹理与默认采样器 fallback
  - `SceneRenderer` 在提交阶段按 `MaterialIndex` 绑定 BaseColor 纹理
- Shader 更新：
  - `model.frag` 增加 `u_BaseColorTex` 采样并与顶点色/光照组合

### 收益

- 最小可用的“带纹理静态网格”链路已经打通。
- 材质贴图绑定仍由 Renderer 主导，保持了 SceneProxy 轻量职责边界。
- 为后续材质系统扩展提供了可演进的基础接口（纹理、采样器、材质槽索引）。

### 新限制

- 当前仅支持 BaseColor 纹理，未接入法线/金属度/粗糙度等材质属性。
- `PipelineKey` 尚未纳入材质维度组合，仍是单 key 通道。
- 纹理绑定仍是提交阶段即时解析，尚未引入 pass 命令缓存。

## 计划中：继续对齐 UE5 的后续方向（待实现）

> 本节为规划项，不代表当前代码已具备这些能力。

### 阶段 4：引入 MeshPassProcessor 形态（待实现）

- 目标：
  - 从“Proxy 直接产出接近提交态的命令”继续推进到“Proxy 产出 MeshBatch 描述，PassProcessor 产生命令”。
- 关键动作：
  - 新增 `FMeshBatch` / `FMeshBatchElement` 级描述类型。
  - `FPrimitiveSceneProxy` 从 `GetMeshDrawCommands` 演进为 `GetMeshBatches` 语义。
  - `SceneRenderer` 引入简化版 `MeshPassProcessor`，统一将 MeshBatch 编译为可提交命令。
- 预期收益：
  - Proxy 与渲染状态组合逻辑进一步解耦。
  - 为材质系统、多 Pass、缓存化命令构建提供稳定入口。

### 阶段 5：引入材质维度的 PipelineKey 组合（待实现）

- 目标：
  - 让 `PipelineKey` 从单一静态网格通道扩展为“Pass + Material + VertexLayout + RenderState”的组合键。
- 关键动作：
  - 将 Pipeline 解析责任从“单 key 映射”升级为“材质/Pass 驱动的 key 生成与缓存命中”。
  - 把共享 Pipeline 缓存逐步迁移到材质系统语义下。
- 预期收益：
  - 与 UE5 “材质 + VertexFactory + Pass”共同决定绘制状态的思路更接近。
  - 消除当前“静态网格单一共享 pipeline”带来的扩展瓶颈。

### 阶段 6：引入 GT/RT 命令队列边界（待实现）

- 目标：
  - 把当前 `IRenderScene` 同步直调路径升级为命令队列消费模型。
- 关键动作：
  - 为 `Add/Update/RemovePrimitive` 建立渲染命令结构与提交队列。
  - 让资源准备、场景注册、命令构建在渲染线程侧执行。
- 预期收益：
  - 线程边界与生命周期管理更贴近 UE5。
  - 为后续可见性并行与命令缓存化提供基础。

### 阶段 7：StaticMesh 资源流送与生命周期细化（待实现）

- 目标：
  - 将 `FStaticMeshRenderData` 从“创建即上传”推进到“可流送、可分级驻留”的资源模型。
- 关键动作：
  - 细化 CPU 描述与 GPU 资源生命周期边界。
  - 引入 LOD 粒度或分块粒度的资源就绪状态。
- 预期收益：
  - 更接近 UE5 静态网格资源流送模型。
  - 降低大场景下的初始化峰值与显存压力。
