# 架构优化复盘：渲染资源准备阶段解耦（第一阶段）

## 背景与问题

在本阶段开始前，`SceneProxy` 虽然已不直接 `CreateBuffer`，但静态网格 GPU 资源准备仍发生在 `TMeshComponent::CreateSceneProxy` 调用链里：
- 组件在创建代理时通过 `IRenderScene` 直接查询 `RenderData/Pipeline`
- `IRenderScene` 同时承担“场景同步接口”和“资源服务接口”
- `FScene` 同时承担场景容器、资源缓存、共享管线初始化多类职责

这会导致职责边界仍然偏混合：代理创建阶段在时序上仍夹带资源准备行为。

## 设计目标

- 让 `CreateSceneProxy` 回归“实例描述创建”语义
- 让资源准备进入独立阶段，并由渲染侧统一触发
- 收敛 `IRenderScene` 边界，只保留场景注册/更新/移除语义
- 将资源管理职责从 `FScene` 抽离到独立管理对象

## 核心方案与取舍

本阶段采用以下方案：
- `IRenderScene` 删除资源查询接口，只保留：
  - `AddPrimitive`
  - `UpdatePrimitiveTransform`
  - `RemovePrimitive`
- `TPrimitiveComponent::CreateSceneProxy` 改为无参数版本；`TMeshComponent::CreateSceneProxy` 仅创建携带 `TStaticMesh` 资产引用的代理
- 新增 `FRenderResourceManager`，集中管理：
  - `TStaticMesh -> FStaticMeshRenderData` 缓存
  - 静态网格共享 Shader/Pipeline
  - 代理资源准备（`PrepareStaticMeshProxy`）
- `FScene::AddPrimitive` 显式调用 `PrepareProxyResources`，资源就绪后才注册 `FPrimitiveSceneInfo`

取舍：
- 当前仍是单线程同步直调，不引入命令队列
- 只处理静态网格代理路径，未扩展到材质系统分层和多渲染路径资源策略

## 收益

- `SceneProxy` 创建阶段不再请求共享 GPU 资源，职责更贴近“实例描述 + 绘制接口”
- 资源准备阶段变得显式：`CreateSceneProxy` 与 `AddPrimitive` 两阶段边界更清晰
- `IRenderScene` 接口语义收敛，`World` 对 `Renderer` 的耦合进一步减小
- `FScene` 职责收敛到场景容器与注册流程，资源细节下沉到 `FRenderResourceManager`

## 当前限制与后续方向

- 仍是单线程同步调用，尚未引入 GT/RT 命令队列
- `FRenderResourceManager` 目前仍在 `Renderer` 模块内，后续可演进为更独立的资源子系统
- Pipeline 仍为静态网格共享通道，尚未按材质/渲染路径细分
- `FStaticMeshRenderData` 仍是“创建即上传”模型，后续可继续拆分 CPU 描述与 GPU 生命周期管理

## 增量记录：第二阶段（2026-04-12）

### 为什么继续改

第一阶段虽然把资源准备从 `CreateSceneProxy` 调用链中剥离出来，但 `FStaticMeshSceneProxy` 仍持有 `RHIPipeline*`，这会让实例镜像层继续携带渲染状态对象，不利于进一步贴近 UE5 的“描述在前、状态解析在后”思路。

### 第二阶段核心改动

- `FMeshDrawCommand` 从直接持有 `RHIPipeline*` 改为持有 `EMeshPipelineKey`。
- `FStaticMeshSceneProxy` 移除 `RHIPipeline*` 成员，只保留：
  - 静态网格资产引用
  - `FStaticMeshRenderData` 引用
  - 产出 `PipelineKey` 的绘制命令
- `FRenderResourceManager::PrepareStaticMeshProxy` 只注入 `RenderData`，并在注册阶段预热共享 Pipeline。
- `SceneRenderer` 在提交阶段通过 `FScene` 解析 `PipelineKey -> Pipeline`，再执行实际绑定与绘制。

### 收益

- SceneProxy 从“资源 + 状态句柄”的混合体进一步收敛为“实例描述 + 绘制描述输出”。
- DrawCommand 语义从“可直接提交的低层状态包”收敛为“渲染侧可解析的中间描述”。
- Pipeline 状态解析集中到 Renderer 侧，更接近 UE5 的 pass/processor 思路。

### 新限制

- 当前 `PipelineKey` 仍是单一静态网格通道，尚未接入材质维度的 key 组合。
- Pipeline 解析仍是单线程即时解析，尚未进入真正的 MeshPass 命令构建缓存体系。
