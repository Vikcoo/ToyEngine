# 项目概览

## 项目定位

ToyEngine 是一个以学习引擎架构为目标的代码库。它借鉴了简化版 UE 风格设计，把游戏侧对象和渲染侧对象分开组织。当前目标不是完整编辑器型引擎，而是一个便于理解运行流和渲染数据组织的小型演示工程。

## 当前已实现内容

当前代码中已经落地的部分：
- 通过 `Platform` 创建窗口并轮询事件，底层实现基于 GLFW
- 通过 `InputManager` 收集键盘、鼠标与滚轮输入
- 存在抽象层 `RHI`，当前实际启用后端为 `OpenGLRHI`
- 通过 Assimp 导入 CPU 侧静态网格资源
- 通过 Actor / Component 体系组织游戏侧场景对象
- 通过 `FScene`、`FPrimitiveSceneProxy`、`FStaticMeshRenderData`、`SceneRenderer` 组织渲染侧数据
- 通过 `Sandbox` 入口创建两个网格 Actor 和一个飞行相机

## 当前未实现或仅占位的内容

以下内容在当前仓库中还没有真正实现：
- 独立渲染线程
- 完整资源注册表或编辑器导入流程
- `VulkanRHI`、`D3D12RHI` 的可运行后端
- 延迟渲染、完整材质系统、灯光系统、多 Pass 渲染器

## 分层视图

```text
Sandbox
  -> Engine
    -> World / Actor / Component 层级（游戏侧状态）
    -> FScene / SceneProxy / StaticMeshRenderData / SceneRenderer（渲染侧状态）
    -> RHI 抽象
    -> OpenGL 后端
    -> Platform + Input + Core 基础设施
    -> Asset 导入与静态网格存储
```

## 核心设计选择

当前项目最重要的架构选择，是显式区分：
- 游戏侧对象：如 `TWorld`、`TActor`、`TPrimitiveComponent`
- 渲染侧对象：如 `FScene`、`FPrimitiveSceneProxy`、`FMeshDrawCommand`

这样做的直接收益是：
- 游戏逻辑不需要直接持有 GPU 资源
- 渲染提交不直接依赖游戏对象内部状态
- 后续如果要引入线程拆分或渲染器重构，边界更清晰

## 可执行入口

当前可执行入口是 `Source/Sandbox/Main.cpp`。  
Sandbox 负责：
- 注册 `Engine::SetSceneSetupCallback`，搭建示例场景
- 注册 `Engine::SetFrameUpdateCallback`，注入示例每帧逻辑
- 调用 `Engine::Get().Init()/Run()/Shutdown()`

因此，`Engine` 负责内核调度，示例内容归属 Sandbox 应用层。
