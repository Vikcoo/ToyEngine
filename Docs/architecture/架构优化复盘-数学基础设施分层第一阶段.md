# 架构优化复盘：数学基础设施分层第一阶段

> 日期：2026-07-14

## 为什么要改

`Core/Math/MathTypes.h` 曾在一个公共头中同时定义向量、矩阵和四元数。对于个人学习引擎，这种集中式入口便于起步，但随着 `Color`、`Random`、`Geometry`、`MathUtils` 等仅需要向量的组件增加，它会让轻量使用者也解析矩阵和四元数定义，依赖边界不够清楚。

同时，视锥提取接口没有表达其手性和深度范围，文档还错误写成 `View * Projection`。在引擎已经统一采用右手系 `[0, 1]` 深度范围后，这种隐式接口会让 NO/ZO 公式混用，Near 平面已因此出现漏剔除风险。

## 本阶段方案与取舍

- 将基础类型拆分为 `Vector.h`、`Matrix.h`、`Quat.h`，各自只包含必要前置依赖。
- 保留 `MathTypes.h` 作为兼容聚合头，现有 Runtime 模块无需因头文件拆分发生一次性迁移；新代码改为按需包含具体头。
- `Transform.h` 显式组合 Matrix 和 Quat；仅使用向量的 `Geometry`、`MathUtils`、`Color`、`Random`、`VectorInt` 改为只依赖 `Vector.h`。
- 将视锥入口命名为 `FromViewProjectionRH_ZO()`，只支持当前引擎标准 `ProjectionRH_ZO * ViewRH`。本阶段不通过布尔参数或运行时分支兼容 LH/NO，避免重新引入隐式约定。

这个阶段的取舍是优先保证 API 语义和编译依赖清晰，而不是立刻将所有外部调用迁移到具体头。`MathTypes.h` 仍保留，降低已有学习代码的迁移成本。

## 本阶段结果

- ZO 视锥 Near 平面改为从 `row2` 提取，匹配裁剪条件 `z >= 0`。
- `Plane(normal, distance)` 现在会将法线和距离作为同一平面方程归一化。
- `RotateLocal*()` 统一使用局部标准轴右乘；世界旋转和局部旋转在组合后都归一化四元数。
- `MathTest` 新增 Plane 方程、ZO Near 平面、局部旋转组合、连续旋转归一化和矩阵分解旋转覆盖。
- 文档已明确 VP 矩阵乘法顺序、深度范围和新的公共头文件职责。

## 收益与限制

收益是数学层的调用依赖更轻，坐标系和深度范围不再依靠调用者记忆，未来新增 LH/NO Frustum 支持时也有明确的扩展模式。

限制仍然存在：`MathTypes.h` 为兼容性继续提供全量包含；`Matrix4::Decompose()` 与 `Transform::FromMatrix()` 仍主要面向无剪切、正缩放 TRS；负缩放、剪切和投影矩阵分解不在本阶段承诺范围内。后续若引入编辑器层级变换或动画导入，应先扩展这些契约与测试。
