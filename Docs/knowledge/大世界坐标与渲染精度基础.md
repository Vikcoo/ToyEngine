# 大世界坐标与渲染精度基础

本文档整理“大世界为什么会丢精度、深度与世界坐标重建是否属于同一个问题、UE5 为什么使用 Translated World Space，以及 ToyEngine 应如何诊断和演进”。目标是建立可复用的精度分析方法，而不是把所有可见误差都笼统归因于深度缓冲。

除非另有说明，文中的数值示例按 `1 个世界单位 = 1 米` 解释。ToyEngine 目前尚未把世界单位写成全局强约定，因此实际误差阈值必须随最终单位制一起确认。

## 先给结论

- 大世界精度问题的根因通常是：用有限精度表示很大的绝对数，同时还要求保留很小的局部变化。数越大，float32 相邻可表示值之间的绝对间隔越大。
- “离世界原点远”和“离相机远”是两个不同维度。前者主要影响绝对世界坐标及其运算，后者主要影响投影、屏幕覆盖率和深度反推。
- 硬件 Depth 保存的是投影后的 `DeviceZ`，不是绝对世界坐标。仅仅把相机和物体一起平移到很远的世界位置，理论上不应改变它们的深度；但如果平移前后的顶点、矩阵已经在 float32 世界坐标中丢失精度，深度仍会被上游误差污染。
- Reversed-Z 主要改善深度缓冲的数值分布，不能解决绝对世界坐标、光源坐标、矩阵平移项或 `大数 - 大数` 的精度问题。
- 屏幕坐标与 Depth 反推位置是延迟渲染的常规方案。重建结果有误差，但不应把“重建位置与 Position RT 的差”全部叫作深度误差，因为两条路径都包含各自的量化、插值和矩阵误差。
- UE5 的通用思路不是让所有 GPU 运算都使用昂贵的高精度数，而是让持久世界坐标具备大范围表达能力，并尽早把渲染运算转换到靠近相机原点的 Translated World Space；只有确实需要绝对世界位置时才继续使用 LWC/DoubleFloat 一类表示。
- ToyEngine 当前的小场景路径是成立的：`D32_Float + Reversed-Z + inverse(ViewProjection)` 可以稳定重建位置。但当前仍在绝对 float32 世界空间中变换和做光照，尚不具备大世界渲染精度架构。

## 需要先区分的三类精度问题

| 问题 | 主要自变量 | 常见现象 | 主要手段 |
| --- | --- | --- | --- |
| 绝对坐标精度 | 距世界原点的数值大小 | 物体抖动、相邻点粘在一起、灯光方向和距离不稳定、世界空间材质跳变 | double、Tile + Offset、Floating Origin、Translated World Space |
| 投影与深度精度 | 距相机距离、Near/Far、投影类型、Depth 格式 | Z-fighting、远处深度量化、重建 ViewZ 或重建位置沿视线方向的误差 | 合理 Near、`D32_Float`、Reversed-Z、分段/局部投影 |
| 空间离散化精度 | 阴影图、体素、距离场、导航网格等单元尺寸 | 阴影游泳、体积跳格、远处 LOD/碰撞粒度不足 | 层级网格、Clipmap、局部坐标、动态分辨率和误差预算 |

三类问题可以同时出现，但不能用同一个修复手段解决。例如 Reversed-Z 能明显改善第二类问题，却不会让一个已经量化到厘米级的绝对世界坐标重新获得毫米级细节。

## float32 为什么在大坐标下丢失局部精度

IEEE 754 float32 有 24 位有效二进制精度，包括隐含的最高位，约等于 7 位十进制有效数字。它提供的是近似固定的相对精度，而不是固定的小数位数。

对普通规格化数，相邻 float32 的间隔可近似写为：

```text
ULP(x) = 2 ^ (floor(log2(abs(x))) - 23)
```

一次正确舍入的误差通常不超过约 `0.5 ULP`，但连续矩阵乘法、加减、插值和归一化会继续传播误差。

按 `1 单位 = 1 米`，典型间隔如下：

| 绝对坐标量级 | 相邻 float32 的间隔约为 | 能否稳定表达 1 mm 变化 |
| --- | ---: | --- |
| 1 m | 0.000000119 m，即 0.119 μm | 可以 |
| 1 km | 0.0000610 m，即 0.061 mm | 可以 |
| 10 km | 0.0009766 m，即 0.977 mm | 已接近极限 |
| 100 km | 0.0078125 m，即 7.81 mm | 不可以 |
| 1000 km | 0.0625 m，即 6.25 cm | 不可以 |

“数太大导致浮点位数减少”是便于理解的说法，更精确的表述是：有效位总数没有减少，但更多有效位被用于表达整数数量级，留给局部小变化的绝对分辨率变粗了。

### 两种最典型的失败

第一种是小增量被大数吞掉：

```cpp
float position = 100000.0f;
position += 0.001f; // 1 mm 小于该量级的 ULP，结果可能完全不变
```

第二种是灾难性消减：

```text
delta = lightAbsolutePosition - surfaceAbsolutePosition
```

数学上 `delta` 可能只有几厘米，但如果两个绝对位置先分别舍入成远离原点的 float32，低位信息已经丢失，之后相减无法恢复。关键原则是：

> 必须在高精度表示仍然保留低位时完成“减去局部原点”的操作，再把小结果降为 float；把两个已经失真的 float 大数拿到 GPU 上相减通常太晚了。

矩阵也受同一规律影响。一个 float32 `LocalToWorld` 若带有很大的平移项，局部顶点与平移相加时就可能抖动；一个包含巨大相机平移的 `ViewProjection` 及其逆矩阵，也可能把消减和求逆误差带入裁剪位置或世界位置重建。

## “深度”实际可能指三种不同的量

讨论误差前必须说明指的是哪一种深度：

1. **DeviceZ / Depth Buffer**：光栅化后写入深度纹理的通常是 `[0,1]` 范围内的投影深度。它一般是非线性的、无单位的。
2. **ViewZ / Linear Depth**：相机空间沿观察方向的有符号坐标或距离，单位与世界单位一致。不同引擎对符号和“线性深度”定义可能不同。
3. **由 Depth 重建的位置**：使用像素 UV、DeviceZ 和逆矩阵得到的 View、Translated World 或 Absolute World Position。它是三维位置，不是深度缓冲本身。

因此，RenderDoc 中看到 `depth = 0.00021`，不能直接解释成 `0.21 mm`。在 ToyEngine 的 Reversed-Z 路径中它是无单位的 DeviceZ；若另一个变量恰好显示 `reconstructionError = 0.00021`，后者才是世界单位长度，两者数值相同只是巧合。

## 离世界原点远与离相机远为什么不同

设相机和表面绝对位置为 `C`、`P`，观察相关计算真正关心的是相对量：

```text
P_view = ViewRotation * (P - C)
```

如果把相机和物体同时平移同一个大偏移 `T`：

```text
(P + T) - (C + T) = P - C
```

数学结果不变，所以 DeviceZ 也应不变。这说明深度天然是相机相对问题，不直接依赖世界原点。

有限精度下，若 `P + T`、`C + T` 或带大平移的矩阵先被存成 float32，再执行相减，等式左右的数值结果可能不再相同。此时画面虽然表现为深度或重建抖动，根因却是上游绝对坐标精度。

反过来，即使相机就在世界原点，物体离相机非常远，透视深度量化、一个像素覆盖的世界尺寸以及逆投影误差仍会增大。这才是“离相机远”的问题。

## 普通 Z、Reversed-Z 与 D32_Float

透视投影的 DeviceZ 大致服从 `a + b / distance`，本身就不是线性分布。普通 Z 把 Near 映射到 0、Far 映射到 1；float 深度在 0 附近的可表示值更密，在 1 附近更稀，因此普通透视分布和浮点分布会共同削弱远处精度。

Reversed-Z 把 Near 映射到 1、Far 映射到 0。配合 `D32_Float` 时，浮点数在 0 附近更密的分布能够较好地抵消透视的 `1 / distance` 分布，所以远距离 DeviceZ 和反推 ViewZ 的精度通常显著改善。

Reversed-Z 仍有边界：

- 它不修复顶点、相机、光源和矩阵在绝对世界空间中已经丢掉的低位。
- 它不修复错误的 UV、Y 翻转、NDC 深度范围、矩阵乘法顺序或跨帧矩阵。
- 它不消除光栅化、采样、矩阵乘法和齐次除法的舍入。
- 极端 Near、深度格式、MSAA 样本选择和深度纹理过滤方式仍需正确配置。

ToyEngine 当前 GPU 主路径已经使用 `D32_Float + Reversed-Z`：Near=1、Far=0、清除值为 0、深度比较为 `Greater`。详细约定见[架构优化复盘：Reversed-Z 深度约定](../architecture/架构优化复盘-Reversed-Z深度约定.md)和[数学坐标系与矩阵 API 约定](../reference/数学坐标系与矩阵API约定.md)。

## 从屏幕与 Depth 重建位置

典型 Deferred 重建流程是：

```text
Pixel/UV
  -> NDC.xy
Depth texture(DeviceZ)
  -> 当前后端约定下的 NDC.z
clip = (NDC.xy, NDC.z, 1)
positionH = InverseViewProjection * clip
position = positionH.xyz / positionH.w
```

如果逆矩阵正是产生该深度的投影矩阵之逆，就应直接使用当前 DeviceZ 约定，不应先随意“线性化”或翻回普通 Z。ToyEngine 当前的 `ReconstructWorldPosition()` 正是直接使用经后端调整的 Reversed-Z `inverse(Projection * View)`。

更适合大世界的变体是先重建 View Position，或直接重建 Translated World Position：

```text
ViewPosition = InverseProjection * clip / w
TranslatedWorldPosition = ViewToTranslatedWorld * ViewPosition
```

这里的 `ViewToTranslatedWorld` 可以只包含旋转和很小的相对平移，避免恢复一个巨大的绝对相机坐标。也可以直接上传 `ClipToTranslatedWorld`，一次矩阵乘法得到相机附近的坐标。

### 重建误差来自哪里

| 环节 | 典型误差或错误 | 降低方式 |
| --- | --- | --- |
| 屏幕坐标 | 像素中心、Viewport 偏移、动态分辨率、TAA Jitter 不一致 | 使用生成 Depth 的同一 ViewRect、Jitter 和像素中心约定 |
| 纹理坐标 | OpenGL/其他后端 Y 方向不同 | 统一 RHI 约定，仅在明确边界翻转一次 |
| NDC.z | `[0,1]` 与 `[-1,1]` 混用，普通 Z 与 Reversed-Z 混用 | 把后端 Depth Convention 显式写入 View 数据和测试 |
| Depth 存储 | D16/D24/D32F 量化、远距离分布、MSAA resolve | `D32_Float + Reversed-Z`，采对样本，合理 Near |
| Depth 采样 | 线性过滤、采到相邻像素、错误 mip | 点采样、mip 0、像素中心；MSAA 时明确 sample |
| 矩阵 | 行列主序、乘法顺序、错误帧、求逆条件数 | 统一矩阵 API，上传与当前 Pass 完全对应的逆矩阵 |
| float 运算 | 矩阵乘法、齐次除法、绝对世界大平移 | 重建到 View/TWS，避免大绝对平移进入高频路径 |
| 对照 Position RT | 顶点世界坐标计算、透视校正 varying 和光栅路径均只有有限精度；RGBA32F 保存的是该路径已经得到的 FP32 近似值 | 不把 Position RT 当作数学真值，分别做单变量对照 |

### ToyEngine 当前 RenderDoc 样本如何解释

此前截取像素中，RenderDoc 显示的截断值约为：

```text
storedWorldPosition        = (0.54713, -0.23214, 0.81441)
reconstructedWorldPosition = (0.54697, -0.23224, 0.81451)
length(error)               ≈ 0.00021 world units
```

若按 `1 单位 = 1 米`，总差约为 `0.21 mm`。至少在该像素上没有出现量级明显的坐标约定错误；若存在 UV 翻转、NDC 范围或矩阵顺序错误，通常会出现远大于亚毫米的系统性偏差。仍应结合整幅 F8 视图、屏幕边缘和多距离样本判断整个重建流程。

但这个 `0.21 mm` 是“两条位置生成路径的总差”，至少包含：

- Position RT 路径的世界坐标计算、varying 插值与光栅化；`RGBA32_Float` 保存的是该路径已经得到的 FP32 近似值，并不因此成为数学真值；
- 重建路径的 UV、Depth 光栅化与 `D32_Float`、采样、float32 逆 VP、矩阵乘法与齐次除法。

因此 F8 适合验证最终重建位置是否稳定，不适合单独证明 Depth 的量化误差是多少。要把问题缩小到 Depth/NDC 链路，可以将 Position RT 重新投影得到参考 DeviceZ，再与 Depth RT 比较，或在 View Space 中比较由同一参考点得到的 ViewZ；但这些结果仍继承 Position RT 的 FP32 计算与插值误差，只是分层诊断量，不是纯深度量化误差。纯量化误差需要受控的高精度解析参考。

## UE5 的 Translated World Space 思路

UE5 具体的底层类型会随版本演进，但稳定的设计原则是：持久/跨系统的绝对位置可以使用更高精度表示，GPU 高频渲染尽早转换到一个靠近当前 View 的小坐标空间。

Translated World Space 的核心关系是：

```text
TranslatedWorldPosition = AbsoluteWorldPosition + PreViewTranslation
AbsoluteWorldPosition   = TranslatedWorldPosition - PreViewTranslation
```

`PreViewTranslation` 通常选择为接近 `-CameraWorldOrigin` 的值，于是相机在 Translated World Space 中接近 `(0,0,0)`，相机附近的物体、灯光和阴影接收点也都是小数值。

这不是必须修改游戏世界中所有 Actor 的“世界原点重置”。它可以是每个 View 独立的渲染转换：权威绝对坐标不变，提交该 View 时只生成相机相对的 GPU 数据。

### 为什么光照适合在 Translated World Space 计算

点光照和观察方向依赖相对向量：

```text
L = normalize(LightPosition - SurfacePosition)
V = normalize(CameraPosition - SurfacePosition)
attenuation = f(length(LightPosition - SurfacePosition))
```

给灯光、表面和相机同时加上 `PreViewTranslation`，数学结果不变：

```text
(Light + T) - (Surface + T) = Light - Surface
```

但有限精度下，先把它们转换成相机附近的小 float 再相减，比让两个巨大 float 绝对坐标相减稳定得多。平移也不会改变法线和方向，因此 BRDF 的 `N·L`、`N·V`、反射方向等结果不变。

Translated World Space 相比纯 View Space 还有一个工程优势：它只移动原点，坐标轴仍保持世界方向。世界对齐材质、重力方向、天空/IBL 方向、场景灯光和许多世界空间资源不需要随相机旋转。View Space 同样能保证相机附近精度，且非常适合屏幕空间、深度分片和 Cluster Culling；但它会旋转坐标轴，需要把更多世界方向和资源同步转换。

所以“UE 的光照在哪个空间”没有一个覆盖所有 Pass 的单一答案，常见分工是：

| 运算 | 常见适合空间 | 原因 |
| --- | --- | --- |
| 通用表面位置、局部灯光与 BRDF | Translated World / Camera-relative World | 保持世界轴，同时让位置靠近原点 |
| 屏幕空间重建、SSAO、Cluster/Tile 划分 | View 或 Screen | 与相机和投影直接对应 |
| Shadow Pass | Light-relative / Shadow View | 关键距离相对于光源或阴影级联 |
| 只依赖方向的 IBL、法线运算 | 一致的 World/TWS 方向基 | 平移不影响方向 |
| 必须跨超大范围访问的绝对世界数据 | LWC/DoubleFloat、Tile + Offset，或局部网格 | 普通 float TWS 无法单独表达全局地址 |

Epic 当前公开的 LWC Rendering 文档描述了两类 GPU 高精度表示：`FDF*` 以 High/Low 两个 float 模拟更高精度，`FLWC*` 以 Tile + Offset 表示位置；高精度运算有明显性能成本，因此官方建议能使用 Translated World Space 时尽早转换并降回普通 float。具体版本实现以 [Epic Large World Coordinates Rendering Overview](https://dev.epicgames.com/documentation/unreal-engine/large-world-coordinates-rendering-in-unreal-engine-5) 为准。

## 大世界还会影响哪些系统

本节列出的是通用引擎风险，不表示 ToyEngine 当前已经实现下列全部系统。

### 渲染

- **顶点变换**：巨大 float 平移与局部顶点相加会让轮廓抖动；网格顶点应保持局部，提交前生成相对 View 的变换。
- **相机与剔除**：Frustum Plane、Bounds、LOD 距离若都用巨大绝对 float，相交测试会在边界闪烁。
- **灯光**：点光位置、距离衰减、灯光体积和 Cluster 边界都会遇到 `大数 - 大数`。
- **法线与导数**：法线本身不受平移影响；但若用巨大 Absolute World Position 的 `ddx/ddy` 计算几何法线或纹理梯度，小差值可能先被吞掉。
- **阴影**：阴影相机矩阵、级联边界、Bias 和世界坐标采样会抖动；通常要使用 Light-relative Space、稳定级联或局部页表。
- **世界空间材质**：对巨大坐标做 `frac`、噪声、三平面映射或周期取模，会因低位不足产生跳变。应先减局部锚点，再做高频细节。
- **Temporal AA 与运动向量**：当前帧和上一帧若使用不同的相对原点，必须把原点变化纳入 Previous Matrix，否则静止物体会产生假速度和拖影。
- **Ray Tracing**：Ray Origin、`Origin + t * Direction`、AABB 和实例平移都可能失精；常见做法是让 BLAS 保持局部，并在受控的相对空间构建/遍历。
- **距离场、体素、雾与 Clipmap**：既受 float 精度影响，也受网格单元尺寸影响，需要分层、局部原点和滚动 Clipmap。

### 非渲染系统

- **物理**：碰撞接触点、约束、速度积分、Broad Phase AABB 和 epsilon 都依赖局部差值。
- **动画与粒子**：Root Motion、骨骼附着、Trail 和长生命周期粒子可能在远处抖动；粒子系统常使用 System-relative 或 Tile-relative Position。
- **导航与 AI**：导航网格单元、路径点、射线查询和距离判断需要区域化，不能只换成 double 就忽略离散分辨率。
- **网络**：位置量化、Replication 范围、分区 ID 和客户端局部原点必须有明确协议；直接发送巨大 float 会丢精度和浪费位宽。
- **音频**：Listener 与 Source 的距离/方向同样应在局部坐标计算。
- **地理坐标**：经纬度/ECEF 数值很大，通常要用 double 保存全局位置，并为渲染与物理建立局部切平面。
- **时间**：这不是空间问题，但属于同一类“大数加小增量”。长期累积的 float 秒数最终无法表达毫秒或帧级变化，应使用 double、整数 Tick 或局部时间基准。

## 解决方案不是只有一种

### CPU double

用 double 保存 Actor、相机、光源和世界变换的权威绝对坐标，能把问题推到远得多的范围。它适合编辑、序列化、网络/地理转换和跨区逻辑，但 GPU 全面使用 FP64 通常成本高、吞吐低，也不能替代局部网格和合理误差预算。

### Tile + Offset 或 High + Low

把位置表示为“大范围索引 + 小范围偏移”，或两个 float 的高低部分。它便于跨超大区域寻址，但运算、比较、插值和矩阵变换必须使用配套函数，不能随意把两部分先合成一个 float。

### Floating Origin / World Origin Rebasing

当玩家远离当前原点时，把场景对象整体减去一个偏移，使权威坐标重新靠近零。它可以让仍使用 float 的物理和 Gameplay 继续工作，但属于全局状态变更，必须同步粒子、物理、导航、音频、网络、缓存和上一帧数据。

### Translated World Space

只在渲染 View 中减去相机或区域原点，不修改权威世界。它成本低、可按 View 独立工作，特别适合 GPU 顶点、Deferred Lighting 和阴影接收点；但需要绝对世界寻址的材质、地理数据或跨区查询仍需额外的 LWC 表示。

成熟引擎通常组合使用这些手段，而不是二选一：CPU/持久数据使用 double 或分区坐标，GPU 高频路径使用 float Translated World Space，少数跨大范围运算使用 DoubleFloat/Tile，超大数据结构再使用分区和 Clipmap。

## 如何诊断误差来源

### 1. 做两个正交实验

**世界原点实验**：保持相机与物体的相对布局完全不变，把相机、物体和灯光一起平移到 `0、1 km、10 km、100 km`。理想情况下 DeviceZ 和画面应不变。若误差随整体偏移增长，优先检查绝对坐标、World/View 矩阵和大数相减。

**相机距离实验**：保持世界原点附近的位置体系不变，只增大相机到物体的相对距离。若误差随距离增长，优先检查 Depth 分布、Near/Far、Depth 格式、像素覆盖率和逆投影。

### 2. 把总位置误差分层

建议依次观察：

1. Position RT 重新投影得到的 DeviceZ 与 Depth RT 的差（只用于缩小到 Depth/NDC 链路，不视为纯深度量化误差）；
2. 重建 View Position 与参考 View Position 的差；
3. View -> Translated World 后的差；
4. 最后才比较 Absolute World Position。

在哪一层突然放大，问题就更可能来自该层的坐标转换，而不是前面所有环节。

### 3. 分解视线方向与横向误差

```glsl
vec3 error = reconstructedPosition - referencePosition;
vec3 rayDirection = normalize(referencePosition - cameraPosition);

float alongRayError = dot(error, rayDirection);
vec3 lateralErrorVector = error - alongRayError * rayDirection;
float lateralError = length(lateralErrorVector);
```

- `alongRayError` 明显较大：优先怀疑 Depth 量化、Depth 约定和投影参数。
- `lateralError` 明显较大：优先怀疑 UV/像素中心、Y 翻转、Jitter、Viewport、矩阵和 Position varying 对照路径。
- 两者在整体远离世界原点时一起增长：优先怀疑绝对 float 世界坐标和逆 VP 中的大平移。

这种分解是定位线索，不是严格的单变量证明，因为逆投影会把输入误差耦合到多个轴。

### 4. RenderDoc 检查清单

- 选中真正执行 Deferred Lighting 的 Draw，而不是 Clear 或 GBuffer Draw。
- 确认采样像素不是几何边缘、背景、透明物或深度不连续处。
- 核对 UV、像素坐标、Viewport 和是否执行 RT Y Flip。
- 查看 Depth 纹理格式、采样器过滤方式、mip、MSAA sample/resolve。
- 核对 Shader 中 NDC 深度范围与 RHI 后端一致。
- 核对当前帧的 Projection、View、Inverse VP 与产生该 Depth 的矩阵完全一致。
- 同时记录 `storedPosition`、`reconstructedPosition`、三轴 error、总长度、ViewZ、DeviceZ、along-ray 与 lateral error。
- 在屏幕中心、四角、近处、远处重复采样，不要用单个像素概括整个误差分布。

## ToyEngine 当前状态与建议演进

### 当前已经实现

- `Vector3`、`Matrix4`、`Transform`、相机位置、Primitive 世界矩阵和灯光位置当前以 float32 为主。
- GPU Forward/GBuffer 使用经后端调整的 Reversed-Z 投影；Deferred Depth 为 `D32_Float`。
- Deferred Lighting 使用采样 Depth 与 `inverse(adjusted Reversed-Z Projection * View)` 重建 **Absolute World Position**。
- 相机和点光以 Absolute World Position 上传，正式 Deferred 光照在绝对 float32 世界空间求 `V`、`L` 和距离衰减。
- `RGBA32_Float` WorldPosition RT 仍由 GBuffer 写入，仅供 F6/F8 对照，正式 Deferred 光照不读取它。
- 当前没有 CPU double 世界坐标、Tile + Offset、World Origin Rebasing、`PreViewTranslation` 或 Translated World 矩阵。

所以当前实现适合现有 Sandbox 的小范围场景，也已经验证了常规屏幕深度重建路径；它不应被描述为已经支持 UE5 式 Large World Coordinates。

### 建议的渐进路线，均为待实现

1. **先建立单位与预算**：正式规定世界单位，定义顶点抖动、重建位置、阴影、物理和网络各自允许的误差，并建立整体平移与相机拉远两组自动化场景。
2. **先引入渲染侧 TWS**：为每个 View 增加 `PreViewTranslation`、`TranslatedWorldToClip`、`ClipToTranslatedWorld`；在仍有较高精度的 CPU 数据上先减去 View Origin，再降为 GPU float。
3. **统一参与位置差的输入**：Primitive 平移、相机、点光、局部光体积和相关 Bounds 必须使用同一个 View Origin；方向和法线只需保持一致的世界方向基。
4. **Deferred 重建到 TWS 或 View**：正式光照不再先恢复巨大 Absolute World Position。屏幕空间算法可使用 View Position，通用表面/灯光可使用 TWS。
5. **加入 Previous Origin**：同时保存当前帧与上一帧的 `PreViewTranslation`，在 Motion Vector/TAA 上线前明确跨帧转换。
6. **为每个 View 独立换基**：主相机、反射、Scene Capture 和阴影视图不能误用另一 View 的相对原点或矩阵。
7. **按目标世界规模升级权威坐标**：若目标超出 float32 预算，再将 CPU Transform/Scene 数据升级为 double 或分区坐标；不要只把 Shader 改成相机相对而让 Gameplay/Physics 继续失真。
8. **最后移除 Position RT**：在纯 Depth、View Space、TWS、屏幕区域和多距离测试都通过后，再删除 WorldPosition attachment、绑定和写带宽。

一个重要实现约束是：如果权威位置当前已经只以远距离 float32 保存，第 2 步无法恢复已经丢掉的低位。要获得真正的大世界能力，减 View Origin 的运算必须发生在 double、Tile/Offset 或其他仍保留低位的表示中。

## 术语速查

| 术语 | 含义 |
| --- | --- |
| Absolute World Space | 以全局世界原点表示的坐标 |
| Translated World Space / Camera-relative World Space | 世界轴方向不变，只把原点平移到 View 附近的坐标 |
| View Space / Camera Space | 原点在相机且坐标轴随相机旋转的坐标 |
| PreViewTranslation | Absolute World -> Translated World 使用的平移量 |
| DeviceZ | 深度缓冲保存的投影深度，通常无单位且非线性 |
| ViewZ / Linear Depth | 相机空间深度或距离，定义与符号需由引擎约定 |
| ULP | 某数值附近相邻可表示浮点数的间隔 |
| Floating Origin | 通过移动权威世界原点让活跃区域靠近零 |
| LWC | Large World Coordinates，大世界坐标体系的统称 |
