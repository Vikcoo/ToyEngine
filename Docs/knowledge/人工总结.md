# 人工总结问题

本文档记录人工写入的问题

## 为什么需要让Device 统一管理 BeginFrame / EndFrame / WaitIdle，Engine 不再持有全局 CommandBuffer 或直接呈现

- 核心原因是：一帧的“获取目标、录制、提交、同步、呈现”属于图形后端生命周期，而不是 Engine 的业务调度，也不只是 CommandBuffer 的录制职责。

## 不再为每次绘制反复覆盖同一块 Uniform Buffer，而是在一块较大的临时 Buffer 中，为每份常量数据分配独立区域。

- 核心原因是：但 Vulkan 的 CommandBuffer 只是记录命令，GPU 可能稍后才执行。Uniform ring 是一块较大的、按顺序分配的 Uniform Buffer，每次上传常量时，只向后申请一段新空间，不覆盖本帧已经分配的数据。 它叫 ring，是因为使用到末尾或经过若干帧后，会重新回到前面的空间继续使用。
- Ring 通常按 frames-in-flight 分段：
  Uniform Ring
  ├─ Frame 0 segment
  ├─ Frame 1 segment
  └─ Frame 2 segment

## dynamic offset 是什么

- 核心原因是：BindGroup/Descriptor 仍然绑定同一个大 Buffer，dynamic offset 告诉 GPU：这次 Draw 应从 Buffer 的哪个位置开始读取。
- 例如：
  Draw A → 绑定 ObjectBlock，dynamicOffset = 0
  Draw B → 绑定 ObjectBlock，dynamicOffset = 256
  Draw C → 绑定 ObjectBlock，dynamicOffset = 512于是三个 Draw 都引用同一 Buffer，但读取不同区域。
## vk和opgl的区别

- 主要差异包括：
  OpenGL 使用全局 binding；Vulkan 使用 set + binding。
  OpenGL 使用 gl_VertexID；Vulkan GLSL 使用 gl_VertexIndex。
  Vulkan 通常要求顶点输入、阶段间变量和 Fragment 输出显式声明 location。
  Vulkan 的普通 Uniform 通常必须放在 Uniform Block 中。
  Vulkan 支持 push_constant，OpenGL GLSL 没有对应语法。
  Vulkan GLSL 编译为面向 Vulkan 环境的 SPIR-V；OpenGL 通常由驱动编译 GLSL 文本。
  两者的 NDC、Y 轴、Framebuffer 原点等约定也存在差异，虽然这部分不完全属于 Shader 语法。
