# 临时架构问题清单

本文档用于记录当前代码中已经确认、但尚未整改的架构问题，供后续分阶段重构时参考。

说明：
- 本文档是临时问题清单，不代表这些问题已经修复。
- 问题按当前影响和后续返工成本排序，不等同于严格的实施顺序。
- 这里优先记录模块边界、依赖方向、生命周期管理、演进阻力等结构性问题，而不是一般代码风格问题。

## 1. RHI 抽象层反向依赖具体后端（已完成第一阶段，2026-04-10）

整改结果（第一阶段）：
- `RHI` 模块已移除对 `OpenGLRHI` 的直接链接和 `OpenGLRHI/Private` 头依赖。
- `RHIFactory.cpp` 已从 `RHI` 模块移除，`RHIDevice::Create()` 的具体装配逻辑已迁移到外层 `Engine` 模块。
- `OpenGLRHI` 新增公开装配入口 `CreateOpenGLRHIDevice()`，避免外层模块包含后端私有头。

剩余事项：
- Vulkan / D3D12 后端尚未接入，后续扩展时需要沿用同样的“后端公开装配入口 + 外层装配”模式。
- 若后续希望继续收敛 `Engine` 责任，可再把后端装配从 `Engine` 抽离到专门的装配模块。

当前相关文件：
- `Source/Runtime/RHI/CMakeLists.txt`
- `Source/Runtime/Engine/Private/RHIDeviceFactory.cpp`
- `Source/Runtime/OpenGLRHI/Public/OpenGLRHIEntry.h`
- `Source/Runtime/OpenGLRHI/Private/OpenGLRHIEntry.cpp`

建议方向：
- 将后端装配逻辑移出 `RHI` 模块本体。
- `RHI` 只保留纯接口、公共枚举、公共描述结构。
- 更外层模块负责选择并创建具体后端，或引入后端注册机制。

## 2. Scene 到 Renderer 的生命周期耦合过重（已完成第一阶段，2026-04-10）

整改结果（第一阶段）：
- `Scene` 引入 `IRenderSceneBridge` 接口，游戏侧不再直接包含/操作 `Renderer` 的 Proxy 类型。
- `TPrimitiveComponent` 改为通过 `BuildRenderCreateInfo` + 句柄进行注册与更新，不再 `new/delete` `FPrimitiveSceneProxy`。
- `TWorld` 改为持有 `IRenderSceneBridge*`，移除了 `FScene*` 和 `RHIDevice*` 的直接持有。
- `FScene` 改为持有 `FPrimitiveSceneProxy` 的所有权（`unique_ptr`），生命周期迁移到渲染侧。

剩余事项：
- 当前桥接层内部仍是直接调用，尚未命令队列化。
- 组件移除与 World 生命周期的统一反注册路径仍可继续收敛（目前已支持组件析构时兜底反注册）。

当前相关文件：
- `Source/Runtime/Scene/CMakeLists.txt`
- `Source/Runtime/Scene/Public/PrimitiveComponent.h`
- `Source/Runtime/Scene/Public/World.h`
- `Source/Runtime/Scene/Public/RenderSceneBridge.h`
- `Source/Runtime/Renderer/Public/FScene.h`
- `Source/Runtime/Renderer/Public/RenderSceneBridgeFactory.h`

建议方向：
- 在桥接接口稳定后，将同步路径从直接调用升级为命令队列消费。
- 继续收敛 `Engine`，把桥接装配下沉到更独立的装配层。

## 3. Engine 同时承担引擎内核和示例应用职责

当前问题：
- `Engine` 既负责底层子系统初始化、主循环和关闭顺序，又直接负责搭建当前示例场景。
- `Engine.cpp` 中包含模型加载、默认立方体生成、相机创建、示例旋转逻辑、FPS 输出等大量产品层或演示层逻辑。
- `Engine` 当前还是单例。

影响：
- 引擎内核和 Sandbox 示例逻辑混杂，复用性差。
- 后续如果想支持多个示例、测试场景、编辑器入口或更干净的应用层，会先被 `Engine` 这一层卡住。
- 单元测试和模块化复用都不方便。

当前相关文件：
- `Source/Runtime/Engine/Public/Engine.h`
- `Source/Runtime/Engine/Private/Engine.cpp`

建议方向：
- `Engine` 只保留子系统装配、主循环与基础调度。
- 把示例场景搭建、演示资源加载、示例每帧逻辑移到 `Sandbox` 或独立应用层。
- 后续再评估是否保留单例模式。

## 4. Platform 层与 OpenGL 上下文初始化绑定过紧

当前问题：
- `GLFWWindow` 构造过程中会直接设置 OpenGL context 版本、创建 OpenGL context 并加载 glad。
- 这使得窗口系统和图形后端初始化被绑定在同一层里。

影响：
- 当前 `Platform` 更像“GLFW + OpenGL 上下文封装”，而不是真正的平台窗口抽象层。
- 后续如果切 Vulkan 或 D3D12，需要重新拆分窗口创建、图形表面创建、函数加载等职责。
- 多后端支持会被平台层结构拖慢。

当前相关文件：
- `Source/Runtime/Platform/Private/GLFW/GLFWWindow.cpp`

建议方向：
- 将“窗口系统”和“图形后端上下文/表面初始化”拆分。
- `Platform` 层尽量只负责窗口、事件、原生句柄暴露等通用能力。
- 具体后端负责自己的上下文或 surface 初始化。

## 5. Scene 与 Renderer 的 PUBLIC 依赖传播过粗

当前问题：
- `Scene` 已收缩为 `Core`、`Asset`、`Input`，但 `Renderer` 仍存在较粗的 `PUBLIC` 依赖传播。
- `Renderer` 当前将 `Core`、`RHI`、`Scene`、`Asset` 作为 `PUBLIC` 依赖。
- 这通常意味着公共头里仍承载了过多跨模块类型。

影响：
- 下游模块更容易被不必要的头文件和类型污染。
- 编译传播面偏大，公共接口变动时更容易触发连锁重编。
- 后续模块边界会越来越难收紧。

当前相关文件：
- `Source/Runtime/Scene/CMakeLists.txt`
- `Source/Runtime/Renderer/CMakeLists.txt`

建议方向：
- 逐步把能前向声明的类型改为前向声明。
- 将仅实现文件需要的依赖从 `PUBLIC` 收缩到 `PRIVATE`。
- 先收窄接口，再收窄链接传播。

## 当前建议的整改顺序

1. 优先把 `Engine` 中的示例应用逻辑拆出。
2. 再清理 `Platform` 职责和模块 `PUBLIC` 依赖传播。
3. 最后把渲染桥接层升级为命令队列模型。
