说明：

- 问题按当前影响和后续返工成本排序，不等同于严格的实施顺序。
- 这里优先记录模块边界、依赖方向、生命周期管理、演进阻力等结构性问题，而不是一般代码风格问题。

## 1. RHI 抽象层反向依赖具体后端（已完成第一阶段，2026-04-10）

整改结果（第一阶段）：
- `RHI` 模块已移除对 `RHIOpenGL` 的直接链接和 `RHIOpenGL/Private` 头依赖。
- `RHIFactory.cpp` 已从 `RHI` 模块移除，`RHIDevice::Create()` 的具体装配逻辑已迁移到外层 `Engine` 模块。
- `RHIOpenGL` 新增公开装配入口 `CreateRHIOpenGLDevice()`，避免外层模块包含后端私有头。

剩余事项：
- Vulkan / D3D12 后端尚未接入，后续扩展时需要沿用同样的"后端公开装配入口 + 外层装配"模式。
- 若后续希望继续收敛 `Engine` 责任，可再把后端装配从 `Engine` 抽离到专门的装配模块。

当前相关文件：
- `Source/Runtime/RHI/CMakeLists.txt`
- `Source/Runtime/Engine/Private/RHIDeviceFactory.cpp`
- `Source/Runtime/RHIOpenGL/Public/RHIOpenGLEntry.h`
- `Source/Runtime/RHIOpenGL/Private/RHIOpenGLEntry.cpp`

建议方向：
- 将后端装配逻辑移出 `RHI` 模块本体。
- `RHI` 只保留纯接口、公共枚举、公共描述结构。
- 更外层模块负责选择并创建具体后端，或引入后端注册机制。

## 2. Scene 到 Renderer 的生命周期耦合过重（已完成第三阶段，2026-04-12）

整改结果（第三阶段）：
- 移除 `RenderSceneBridge` 路径，改为由 `FScene` 直接实现 `IRenderScene` 接口。
- `World` 通过 `IRenderScene*` 与渲染侧交互，`Engine` 不再装配额外桥接对象。
- `FStaticMeshSceneProxy` 不再创建 GPU 资源，也不再依赖 `RHIDevice*`。
- 新增 `FStaticMeshRenderData` 资产级共享资源，统一持有静态网格的 GPU Buffer。
- 相同 `StaticMesh` 的多个实例共享同一份 RenderData，避免重复上传网格数据。
- Shader / Pipeline 生命周期从 `SceneProxy` 下沉到 `FScene` 统一管理与复用。
- `PrimitiveComponent` 回归 `CreateSceneProxy` 语义，移除 `BuildRenderCreateInfo` 命名路径。
- 移除 `RenderPrimitiveHandle`，同步改为按 `FPrimitiveComponentId` 注册、更新、销毁。
- `IRenderScene` 接口调整为 `Add/Update/RemovePrimitive`，贴近 UE5 的调用语义。

剩余事项：
- `IRenderScene` 当前仍是同步直接调用，尚未命令队列化。
- 材质系统仍是简化版，Pipeline 缓存尚未按材质维度拆分。
- `RHICommandBuffer` 仍没有 `TransitionResource` 等显式资源屏障接口；OpenGL 依赖隐式同步可工作，但 Vulkan / D3D12 后端接入前必须补齐。
- Forward / Deferred 主路径已引入 `BindGroupLayout` / `PipelineLayout`，并将 group index 与 shader binding slot 拆分为 `RendererBindGroups` / `RendererBindings`；但 shader `layout(binding = N)` 与 C++ layout 声明仍是两份真相源。
- Renderer 已改为只请求逻辑 Shader 名称，具体 OpenGL GLSL 路径由 `OpenGLShaderLibrary` 解析；但映射和资源布局仍为手写，GLSL -> SPIR-V / DXIL、跨后端反射与自动资产清单尚未建立。
- `FRenderResourceManager` 已从 `FScene` 拆出资源准备职责，但两者当前仍在同一模块内，后续可继续演进为更清晰的渲染资源子系统。
- 当前已引入 `FPrimitiveComponentId` 作为主索引并新增 `FPrimitiveSceneInfo`，但仍是单线程同步调用，后续进入 GT/RT 时仍需命令队列边界。
- 渲染命令构建已在前向/延迟双路径 Stage 2 引入最小 `FMeshPassProcessor(BasePass)`，但仍未引入显式 `MeshBatch` 与多 Pass 命令缓存。

当前相关文件：
- `Source/Runtime/World/Public/RenderScene.h`
- `Source/Runtime/World/Public/PrimitiveComponent.h`
- `Source/Runtime/World/Private/PrimitiveComponent.cpp`
- `Source/Runtime/World/Public/World.h`
- `Source/Runtime/World/Private/World.cpp`
- `Source/Runtime/RenderCore/Public/PrimitiveSceneProxy.h`
- `Source/Runtime/RenderCore/Public/PrimitiveComponentId.h`
- `Source/Runtime/RenderCore/Public/PrimitiveSceneInfo.h`
- `Source/Runtime/RenderCore/Public/LightComponentId.h`
- `Source/Runtime/RenderCore/Public/LightSceneProxy.h`
- `Source/Runtime/RenderCore/Public/StaticMeshRenderData.h`
- `Source/Runtime/RenderCore/Public/StaticMeshSceneProxy.h`
- `Source/Runtime/Renderer/Public/RendererScene.h`
- `Source/Runtime/Renderer/Private/RendererScene.cpp`

建议方向：
- 以 `IRenderScene` 为边界引入渲染命令队列。
- 将 Pipeline/Shader 缓存继续收敛到材质系统层。
- 在 `RHICommandBuffer` 增补资源状态切换接口，并保证 OpenGL/Vulkan/D3D12 语义对齐。
- 建立正式的 SPIR-V shader 编译与反射流程，避免多后端继续依赖手工分发 shader 资产和手写 binding layout。
- 把 `FScene` 中的资源管理职责逐步下沉到独立 `RenderResource` 子系统。
- 参考 `Docs/architecture/架构优化复盘-渲染资源准备阶段解耦第一阶段.md` 中"计划中：继续对齐 UE5 的后续方向（待实现）"按阶段推进。

## 3. Engine 同时承担引擎内核和示例应用职责（已完成第二阶段，2026-06-16）

整改结果（第一阶段）：
- `Engine` 移除了示例层逻辑（模型导入、默认立方体生成、示例旋转），仅保留内核装配、主循环、渲染调度与关闭顺序。
- `Engine` 新增应用层注入接口：`SetSceneSetupCallback`、`SetFrameUpdateCallback`、`SetActiveCameraComponent`。
- `Sandbox/Main.cpp` 接管示例场景搭建和示例每帧逻辑，`Engine` 不再直接构建示例内容。

整改结果（第二阶段）：
- `Engine::Tick` 拆为显式单线程帧阶段：`PumpPlatformMessages`、`TickInput`、`TickGameThread`、`SendAllEndOfFrameUpdates`、`TickRenderThread`、`EndFrame`。
- `SendAllEndOfFrameUpdates` 成为游戏侧状态提交到渲染侧的明确边界，当前仍直接调用 `World::SyncToScene()`。
- `TickGameThread` / `TickRenderThread` 只是职责阶段命名，当前没有引入独立线程或完整 Tick 调度系统。

剩余事项：
- `Engine` 仍采用单例模式，后续可在多运行实例/测试隔离需求下继续评估拆分。
- 当前回调仍是函数式注入，后续可演进为更明确的 `Application` 接口对象。
- Actor / Component 仍是简单遍历 Tick，尚未引入 UE 风格 TickGroup。

当前相关文件：
- `Source/Runtime/Engine/Public/Engine.h`
- `Source/Runtime/Engine/Private/Engine.cpp`
- `Source/Sandbox/Main.cpp`
- `Source/Runtime/Engine/CMakeLists.txt`
- `Source/Sandbox/CMakeLists.txt`

建议方向：
- 在保持 `Engine` 内核职责稳定的前提下，继续把不同示例拆分为独立应用入口。
- 评估引入统一的应用生命周期接口（Startup/Update/Shutdown）以替代散落回调。

## 4. Platform 层与 OpenGL 上下文初始化绑定过紧（已完成第一阶段，2026-04-16）

## 5. World 与 Renderer 的 PUBLIC 依赖传播过粗

当前问题：
- `World` 已收缩为 `Core`、`Asset`、`Input`，但 `Renderer` 仍存在较粗的 `PUBLIC` 依赖传播。
- `Renderer` 当前将 `Core`、`RHI`、`World`、`Asset` 作为 `PUBLIC` 依赖。
- 这通常意味着公共头里仍承载了过多跨模块类型。

影响：
- 下游模块更容易被不必要的头文件和类型污染。
- 编译传播面偏大，公共接口变动时更容易触发连锁重编。
- 后续模块边界会越来越难收紧。

当前相关文件：
- `Source/Runtime/World/CMakeLists.txt`
- `Source/Runtime/Renderer/CMakeLists.txt`

建议方向：
- 逐步把能前向声明的类型改为前向声明。
- 将仅实现文件需要的依赖从 `PUBLIC` 收缩到 `PRIVATE`。
- 先收窄接口，再收窄链接传播。

## 6. PBR 与 IBL 后续质量问题（材质绑定与 IBL 初版已完成，2026-06-18）

当前问题：
- 资产侧 `FMaterial`、材质 GPU 绑定、PBR 贴图采样、Tangent 导入和路径级纹理缓存已完成基础接入。
- RHI / OpenGL 已支持 TextureCube，Renderer 已能从 HDR equirectangular 图运行时生成 IBL 初版。
- 当前 specular IBL 仍使用环境 cubemap mip 链近似，不是正式 GGX prefiltered specular cubemap。
- IBL 预计算为 CPU 运行时计算，尚未缓存落盘，也没有环境资源选择/热重载流程。
- 天空背景只有局部 Reinhard 映射，项目仍缺正式曝光与 Tonemapping。
- PBR 参数调试视图尚未建立，后续只看最终 Lit 结果会增加排障成本。

影响：
- 当前 IBL 能让物体从 cubemap 获得环境贡献，但高粗糙金属、强反射材质的结果还不能作为最终物理质量判断。
- 首次加载 HDR 时会有 CPU 预计算成本，后续场景切换或多环境资源时会变成明显卡顿点。
- 后续接入 glTF PBR 时，Assimp 材质信息若没有先转换为引擎自有描述，会继续扩大 Asset / Renderer 耦合。
- 没有 PBR 参数 DebugView 时，AO 贴图全黑、Metallic 错误、Normal 方向错误、IBL 过曝/欠曝仍然需要靠改 shader 排查。

当前相关文件：
- `Source/Runtime/Asset/Public/Material.h`

- `Source/Runtime/Asset/Private/AssetImporter.cpp`
- `Source/Runtime/RenderCore/Public/StaticMeshRenderData.h`
- `Source/Runtime/Renderer/Private/RenderResourceManager.cpp`
- `Source/Runtime/Renderer/Private/ForwardRenderPath.cpp`
- `Source/Runtime/Renderer/Private/DeferredRenderPath.cpp`

- `Source/Runtime/RHIOpenGL/Private/OpenGLTexture.cpp`
- `Docs/architecture/PBR渲染前置规划.md`

建议方向：
- 将 specular prefilter 升级为 GGX 重要性采样预滤波 cubemap，保留当前 Environment BindGroup 形状。
- 为 IBL 预计算结果增加磁盘缓存或资源系统缓存，避免每次运行重复计算。
- 增加曝光、Tonemapping 和 PBR/IBL DebugView。
- 把固定 HDR 路径升级为场景级 SkyLight / Environment 配置。

## 7. Core 数学旋转约定曾未完全收敛（已修复当前测试暴露项）

历史问题：
- `MathTest` 曾暴露 `Quat::FromEuler()` / `ToEulerAngles()` 往返不一致。
- 旧 `Transform::LookAt()` 生成的旋转曾与 `GetForward()` 的局部前向轴约定不一致。
- 这些问题会影响相机、光源方向、调试几何朝向和未来编辑器 gizmo 的基础可信度。

整改结果：
- `Quat::ToEulerAngles()` 已对齐 `Quat::FromEuler()` 的 `yaw(Y) * pitch(X) * roll(Z)` 约定。
- `Transform::SetForwardRH()`、成员 `LookAtRH()`、静态 `LookAtRH()` 已统一使用普通 Transform 局部 `+Z` 前向和右手正交基。
- `Matrix4` View/Projection 公共 API 已显式区分 `RH/LH` 手性和 `ZO/NO` 深度范围，当前相机使用 `LookAtRH()` 与 `PerspectiveRH_ZO()`。
- `MathTest` 已补充 yaw/roll 往返、`LookAtRH +X` 和 `SetForwardRH -Z` 覆盖，Release 下已全部通过。
- `Frustum` 已收敛为显式 `FromViewProjectionRH_ZO()` 入口，避免以同一 API 隐式兼容不同深度范围；`MathTest` 已覆盖 ZO Near 平面。
- 数学类型已从单一 `MathTypes.h` 拆为 `Vector.h`、`Matrix.h`、`Quat.h`，并保留 `MathTypes.h` 作为兼容聚合头，降低轻量数学使用者的编译依赖。

剩余影响：
- 相机仍采用局部 `-Z` 为看向前方，该差异需要继续限制在 `CameraComponent` 内。
- 后续接入编辑器、动画或序列化前，仍应扩展更多轴向与组合旋转测试。

## 8. 内存系统 Shutdown 并发边界曾不清晰（已按方案 A 收敛）

历史问题：
- `SimpleTest` 的 `TestShutdownRaceNoCrash()` 曾可触发访问冲突。
- `MemoryShutdown()` 可以在其它线程仍持有由 `MemAlloc()` 返回的裸指针时销毁 allocator 和底层 pool。
- `MemFree()` 只能查看当前全局 allocator，无法可靠处理来自旧 allocator 的指针。

整改结果：
- 已明确采用方案 A：`MemoryShutdown()` 是有序停机接口，不支持与 `MemAlloc()` / `MemFree()` 并发调用。
- `Memory.h` 已写明关闭前置条件：所有使用内存系统的 worker 必须停止，旧裸指针必须释放。
- `SimpleTest` 已改为 `TestOrderedShutdownNoCrash()`，按 stop worker、join worker、shutdown 的顺序验证。
- Release 下 `SimpleTest.exe` 已通过。

剩余限制：
- 当前实现仍不支持热重载式或抢占式并发 shutdown。
- 如果未来需要方案 B，仍需引入 allocator/pool 归属、关闭状态、活跃操作计数或延迟释放等机制。

当前相关文件：
- `Source/Runtime/Core/Private/Memory/Memory.cpp`
- `Source/Runtime/Core/Private/Memory/TlsfAllocator.cpp`
- `Tests/SimpleTest.cpp`
