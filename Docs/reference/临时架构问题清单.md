# 临时架构问题清单

本文档用于记录当前代码中已经确认、但尚未整改的架构问题，供后续分阶段重构时参考。

说明：
- 本文档是临时问题清单，不代表这些问题已经修复。
- 问题按当前影响和后续返工成本排序，不等同于严格的实施顺序。
- 这里优先记录模块边界、依赖方向、生命周期管理、演进阻力等结构性问题，而不是一般代码风格问题。

## 1. RHI 抽象层反向依赖具体后端（已完成第一阶段，2026-04-10）

整改结果（第一阶段）：
- `RHI` 模块已移除对 `RHIOpenGL` 的直接链接和 `RHIOpenGL/Private` 头依赖。
- `RHIFactory.cpp` 已从 `RHI` 模块移除，`RHIDevice::Create()` 的具体装配逻辑已迁移到外层 `Engine` 模块。
- `RHIOpenGL` 新增公开装配入口 `CreateRHIOpenGLDevice()`，避免外层模块包含后端私有头。

剩余事项：
- Vulkan / D3D12 后端尚未接入，后续扩展时需要沿用同样的"后端公开装配入口 + 外层装配"模式。
- 若后续希望继续收敛 `Engine` 责任，可再把后端装配从 `Engine` 抽离到专门的装配模块。

当前相关文件：
- `Source/Runtime/RHI/CMakeLists.txt`
- `Source/Runtime/Engine/Private/RHIDeviceFactory.cpp`
- `Source/Runtime/RHIOpenGL/Public/RHIOpenGLEntry.h`
- `Source/Runtime/RHIOpenGL/Private/RHIOpenGLEntry.cpp`

建议方向：
- 将后端装配逻辑移出 `RHI` 模块本体。
- `RHI` 只保留纯接口、公共枚举、公共描述结构。
- 更外层模块负责选择并创建具体后端，或引入后端注册机制。

## 2. Scene 到 Renderer 的生命周期耦合过重（已完成第三阶段，2026-04-12）

整改结果（第三阶段）：
- 移除 `RenderSceneBridge` 路径，改为由 `FScene` 直接实现 `IRenderScene` 接口。
- `TWorld` 通过 `IRenderScene*` 与渲染侧交互，`Engine` 不再装配额外桥接对象。
- `FStaticMeshSceneProxy` 不再创建 GPU 资源，也不再依赖 `RHIDevice*`。
- 新增 `FStaticMeshRenderData` 资产级共享资源，统一持有静态网格的 GPU Buffer。
- 相同 `TStaticMesh` 的多个实例共享同一份 RenderData，避免重复上传网格数据。
- Shader / Pipeline 生命周期从 `SceneProxy` 下沉到 `FScene` 统一管理与复用。
- `TPrimitiveComponent` 回归 `CreateSceneProxy` 语义，移除 `BuildRenderCreateInfo` 命名路径。
- 移除 `RenderPrimitiveHandle`，同步改为按 `TPrimitiveComponent*` 直接注册、更新、销毁。
- `IRenderScene` 接口调整为 `Add/Update/RemovePrimitive`，贴近 UE5 的调用语义。

剩余事项：
- `IRenderScene` 当前仍是同步直接调用，尚未命令队列化。
- 材质系统仍是简化版，Pipeline 缓存尚未按材质维度拆分。
- `FRenderResourceManager` 已从 `FScene` 拆出资源准备职责，但两者当前仍在同一模块内，后续可继续演进为更清晰的渲染资源子系统。
- 当前已引入 `FPrimitiveComponentId` 作为主索引并新增 `FPrimitiveSceneInfo`，但仍是单线程同步调用，后续进入 GT/RT 时仍需命令队列边界。
- 渲染命令构建已在前向/延迟双路径 Stage 2 引入最小 `FMeshPassProcessor(BasePass)`，但仍未引入显式 `MeshBatch` 与多 Pass 命令缓存。

当前相关文件：
- `Source/Runtime/World/Public/RenderScene.h`
- `Source/Runtime/World/Public/PrimitiveComponent.h`
- `Source/Runtime/World/Private/PrimitiveComponent.cpp`
- `Source/Runtime/World/Public/World.h`
- `Source/Runtime/World/Private/World.cpp`
- `Source/Runtime/RenderCore/Public/PrimitiveSceneProxy.h`
- `Source/Runtime/RenderCore/Public/PrimitiveComponentId.h`
- `Source/Runtime/RenderCore/Public/PrimitiveSceneInfo.h`
- `Source/Runtime/RenderCore/Public/StaticMeshRenderData.h`
- `Source/Runtime/RenderCore/Public/StaticMeshSceneProxy.h`
- `Source/Runtime/Renderer/Public/RendererScene.h`
- `Source/Runtime/Renderer/Private/RendererScene.cpp`

建议方向：
- 以 `IRenderScene` 为边界引入渲染命令队列。
- 将 Pipeline/Shader 缓存继续收敛到材质系统层。
- 把 `FScene` 中的资源管理职责逐步下沉到独立 `RenderResource` 子系统。
- 参考 `Docs/architecture/架构优化复盘-渲染资源准备阶段解耦第一阶段.md` 中"计划中：继续对齐 UE5 的后续方向（待实现）"按阶段推进。

## 3. Engine 同时承担引擎内核和示例应用职责（已完成第一阶段，2026-04-11）

整改结果（第一阶段）：
- `Engine` 移除了示例层逻辑（模型导入、默认立方体生成、示例旋转），仅保留内核装配、主循环、渲染调度与关闭顺序。
- `Engine` 新增应用层注入接口：`SetSceneSetupCallback`、`SetFrameUpdateCallback`、`SetActiveCameraComponent`。
- `Sandbox/Main.cpp` 接管示例场景搭建和示例每帧逻辑，`Engine` 不再直接构建示例内容。

剩余事项：
- `Engine` 仍采用单例模式，后续可在多运行实例/测试隔离需求下继续评估拆分。
- 当前回调仍是函数式注入，后续可演进为更明确的 `Application` 接口对象。

当前相关文件：
- `Source/Runtime/Engine/Public/Engine.h`
- `Source/Runtime/Engine/Private/Engine.cpp`
- `Source/Sandbox/Main.cpp`
- `Source/Runtime/Engine/CMakeLists.txt`
- `Source/Sandbox/CMakeLists.txt`

建议方向：
- 在保持 `Engine` 内核职责稳定的前提下，继续把不同示例拆分为独立应用入口。
- 评估引入统一的应用生命周期接口（Startup/Update/Shutdown）以替代散落回调。

## 4. Platform 层与 OpenGL 上下文初始化绑定过紧（已完成第一阶段，2026-04-16）

整改结果（第一阶段）：
- `FWindowConfig` 新增 `graphicsAPI` 字段（`EWindowGraphicsAPI::OpenGL` / `None`），窗口创建时根据此字段决定是否初始化 OpenGL Context。
- `GLFWWindow` 在 `graphicsAPI == None` 时使用 `GLFW_NO_API` hint，不创建 OpenGL Context，不加载 glad。
- `IWindow::SwapBuffers()` / `SetVSync()` / `IsVSyncEnabled()` 改为带默认空实现的虚方法（非 OpenGL 后端无需实现）。
- 新增 `IWindow::GetGraphicsAPI()` 查询方法。

剩余事项：
- `SwapBuffers` 仍保留在 `IWindow` 接口上，Vulkan/D3D12 后端的呈现逻辑尚未抽象。
- 后续可在 `RHIDevice` 上引入统一的 `Present()` 或 `SwapChain` 抽象。

当前相关文件：
- `Source/Runtime/Platform/Public/Window.h`
- `Source/Runtime/Platform/Private/GLFW/GLFWWindow.h`
- `Source/Runtime/Platform/Private/GLFW/GLFWWindow.cpp`

## 5. World 与 Renderer 的 PUBLIC 依赖传播过粗

当前问题：
- `World` 已收缩为 `Core`、`Asset`、`Input`，但 `Renderer` 仍存在较粗的 `PUBLIC` 依赖传播。
- `Renderer` 当前将 `Core`、`RHI`、`World`、`Asset` 作为 `PUBLIC` 依赖。
- 这通常意味着公共头里仍承载了过多跨模块类型。

影响：
- 下游模块更容易被不必要的头文件和类型污染。
- 编译传播面偏大，公共接口变动时更容易触发连锁重编。
- 后续模块边界会越来越难收紧。

当前相关文件：
- `Source/Runtime/World/CMakeLists.txt`
- `Source/Runtime/Renderer/CMakeLists.txt`

建议方向：
- 逐步把能前向声明的类型改为前向声明。
- 将仅实现文件需要的依赖从 `PUBLIC` 收缩到 `PRIVATE`。
- 先收窄接口，再收窄链接传播。
