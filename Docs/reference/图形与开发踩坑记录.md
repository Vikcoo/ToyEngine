# 图形与开发踩坑记录

本文档用于长期沉淀本仓库在图形学实现、运行时架构、工具链使用过程中已经遇到过的 bug、误判、实现陷阱与仍需警惕的潜在问题。

它和 [`临时架构问题清单`](./临时架构问题清单.md) 的区别是：
- `临时架构问题清单` 更偏“尚未整改的结构性问题”
- 本文档更偏“已经踩过或明确识别过的坑”，包括已修复问题、定位经验、以及值得反复提醒的工程误区

目标：
- 给后续开发提供可复用的排障经验
- 把“为什么会错”记录下来，而不仅是“后来怎么改”
- 帮助学习图形学与引擎开发中的真实实现细节

## 维护规则

- 每次出现明确 bug，修复时都应在本文档补一条记录。
- 每次识别出虽然暂未触发、但后续高概率踩中的潜在问题，也应补一条记录。
- 记录必须至少说明：
  - 现象或风险是什么
  - 根因是什么
  - 最终如何处理，或为什么暂时不处理
  - 后续再遇到时应优先检查什么
- 若某条记录后来被新的实现覆盖，不删除旧记录，而是在原条目下补“后续状态”。

## 一、已修复或已明确定位的坑

### 1. Deferred RT 采样 Y 翻转误判

- **现象**：切到 Deferred 后整张画面上下颠倒，鼠标抬头/低头视觉上也反了，光照方向看起来不对。
- **根因**：Lighting Pass 里对 RT 采样又做了一次 `Y` 翻转，但当前 OpenGL 全屏三角形路径的 UV 已经和 FBO 纹理坐标匹配，不需要再翻。
- **处理**：将 `RHIBackendTraits::bRTSampleRequiresFlipY` 在当前 OpenGL 路径上设为 `false`，并保留该 trait 作为后端约定显式入口。
- **经验**：RT 采样坐标是否需要翻转，必须作为“后端约定”显式管理，不能凭印象在 shader 里硬编码。
- **优先检查项**：`RHIBackendTraits`、全屏三角形 UV 生成方式、FBO 纹理原点约定。

### 2. Deferred Lighting 误用默认线性采样器

- **现象**：Deferred 路径下，斜着看某个面时异常变亮，正着看又变暗。
- **根因**：Lighting Pass 复用了默认 `Linear` sampler 去采样 GBuffer。BaseColor 线性过滤通常可以接受，但 Normal 和尤其是 Depth 不能被线性插值，否则会破坏延迟渲染输入。
- **处理**：新增专用 `GBuffer_Nearest_Sampler`，统一对 GBuffer 与 Depth 使用 `Nearest + ClampToEdge`。
- **经验**：GBuffer 不是普通颜色贴图。法线、深度、材质参数、ID 这类“结构化数据”默认都应按离散值处理，不能先假定线性过滤安全。
- **优先检查项**：Lighting Pass 的 sampler 绑定、法线编码、Depth 采样方式。

### 3. Deferred 依赖深度重建世界坐标导致点光视角相关误差

- **现象**：即使修正了 GBuffer 采样器，Deferred 下同一表面的点光亮度仍随观察角度明显变化。
- **根因**：当前 OpenGL 深度约定、投影矩阵适配和 `inverse(ViewProjection)` 重建链路之间仍存在不一致，导致 `Depth -> WorldPosition` 反推结果不稳定。
- **处理**：GBuffer 新增 `WorldPosition` 通道，Lighting Pass 直接读取世界坐标，不再依赖深度重建。
- **代价**：额外增加一张 `RGBA32_Float` RT，带宽和显存成本上升。
- **经验**：当多后端深度约定尚未完全校准时，位置重建很容易成为隐性误差源。学习阶段先用显式 Position RT 保证正确性是合理取舍。
- **优先检查项**：NDC 深度范围、`AdjustProjectionMatrix()`、深度纹理采样值、重建公式与 API 约定是否一致。
- **后续状态**：已增加 `WorldPositionReconstructionError` 调试视图，并将 GPU 主渲染路径切换为 Reversed-Z。正式 Deferred Lighting 现已改用 Depth 与 `inverse(adjusted Reversed-Z Projection * View)` 的重建位置；Position RT 暂时保留给 `F6/F8` 对照。若重新出现点光随视角漂移，应优先用 F8 检查重建误差，再核对 Reversed-Z 投影、深度背景值、NDC 范围与 RT Y 翻转。

### 4. 普通浮点深度导致远处世界坐标重建误差放大

- **现象**：`F8` 误差视图中，靠近模型时方块多为蓝色；相机后退后逐渐变为青、绿、黄，表示误差从微米级增长到毫米级。
- **根因**：普通 Z 将 Near 映射到 0、Far 映射到 1。透视深度已经偏向近处，`D32_Float` 又在 0 附近更密、1 附近更稀，两种分布叠加后远处深度量化误差明显；逆 VP 的浮点运算还会进一步放大世界空间绝对误差。
- **处理**：Renderer 在 GPU 提交边界把正向 ZO 投影转换为 Reversed-Z，统一使用 Near=1、Far=0、清除值 0 和 `Greater` 比较；Deferred 背景判定同步改为深度接近 0。CPU Frustum 保持正向 RH_ZO。
- **经验**：Reversed-Z 不增加位数也不线性化深度，它利用浮点数在 0 附近更密集的分布改善远处精度。投影、清除值、比较函数、背景判定、深度重建必须作为一组同时修改。
- **优先检查项**：GPU MVP 是否经过 `RendererDepthConvention`、深度是否清 0、Pipeline 是否使用 `Greater`、天空是否判断深度接近 0、inverse VP 是否来自同一 Reversed-Z 投影。
- **后续状态**：实测 `F8` 总误差颜色没有明显下降。float32 数值复算确认 Reversed-Z 已把当前相机范围内的投影/深度重建误差从毫米级压到微米级，因此剩余颜色主要由 F8 同时包含的 varying 插值、光栅精度、Position RT 路径和世界空间逆变换等误差贡献。下次应先增加纯深度误差视图，不要把 F8 当成 Reversed-Z 的单变量测试。

### 5. Forward / Deferred 切换后视觉上可能完全一致

- **现象**：按 `F1` / `F2` 切换时，画面看起来几乎没有区别。
- **根因**：两条路径当前使用的是同一套基础光照公式，只是执行位置不同：Forward 在 BasePass 中算，Deferred 在 Lighting Pass 中算。
- **处理**：保留该行为，不视为 bug。
- **经验**：渲染路径切换不等于着色模型切换。若要验证路径切换，应增加调试视图、路径提示或统计信息，而不是用“肉眼看起来应该不同”判断。
- **优先检查项**：日志是否打印路径切换、Shader 公式是否本来就等价、是否存在调试视图。

### 6. Sandbox 可执行文件占用导致 Windows 链接失败

- **现象**：构建走到最后链接 `bin/Sandbox.exe` 时失败，提示 `Permission denied` 或拒绝访问。
- **根因**：Windows 下正在运行的可执行文件会锁定目标输出，链接器无法覆盖。
- **处理**：关闭正在运行的 `Sandbox.exe` 后重新构建。
- **经验**：这种失败首先归类为“运行中的产物占用”，不要一上来怀疑源码或链接参数。
- **优先检查项**：是否仍开着 Sandbox、任务管理器里的进程状态、输出文件占用情况。

### 7. CMake / g++ 调用失败不一定是源码问题

- **现象**：`cmake --build` 或 `g++.exe` 无法正常运行。
- **根因**：CLion 自带 MinGW 工具链的 `PATH` 未注入当前进程，或者当前 PowerShell 环境不满足该调用方式。
- **处理**：优先补齐 `C:\Program Files\JetBrains\CLion 2025.3.3\bin\mingw\bin` 到 `PATH`，使用仓库约定的 CLion CMake 可执行文件，再判断是否为代码问题。
- **经验**：工具链环境未校准前，不要把“无法构建”归因到源码。
- **优先检查项**：`PATH`、CMake 可执行文件路径、构建目录、当前 shell 权限和环境。

### 8. 法线不能直接用 `mat3(model)` 变换

- **现象**：在存在非等比缩放的模型上，光照方向会悄悄出错，表现为亮面和暗面的分布不符合几何直觉。
- **根因**：法线不是普通位置向量，不能直接乘 `mat3(u_Model)`；正确做法是使用模型矩阵左上角 `3x3` 的逆转置。
- **处理**：Forward / Deferred 顶点 shader 统一改为使用 `u_NormalMatrix`，CPU 侧用 `WorldMatrix.GetNormalMatrix()` 上传。
- **经验**：只要涉及光照，就不要默认“位置矩阵也能直接变法线”。等比缩放时看起来没问题，不代表公式是对的。
- **优先检查项**：shader 里的 normal transform、CPU 上传的 normal matrix、是否存在非等比缩放。

### 9. 路径切换需要独立的调试反馈

- **现象**：Forward / Deferred 使用同一套光照公式时，切换后肉眼可能几乎看不出区别，容易误判成切换没生效。
- **根因**：渲染路径差异在于 Pass 组织方式，不一定体现在最终颜色公式上。
- **处理**：为 Sandbox 增加窗口标题状态显示和 Deferred GBuffer 调试视图快捷键（当前为 `F3-F8`）。
- **经验**：做渲染系统时，验证能力本身也是功能的一部分；没有 debug view，很容易把“看起来没变化”误认为 bug。
- **优先检查项**：当前路径状态、调试视图模式、日志是否有切换记录。

### 10. OpenGL 4.1 不能直接在 UBO 上写 `layout(binding = N)`

- **现象**：shader 编译失败，错误类似：`layout qualifier 'binding' requires "#version 420" or later`。
- **根因**：问题出现时项目目标还是 OpenGL 4.1 Core；`layout(binding = N)` 这类 uniform block binding 语法需要 GLSL 4.20 或 `GL_ARB_shading_language_420pack`。
- **处理**：shader 保留 `layout(std140) uniform LightBlock`，不在 GLSL 中写 binding；Program 链接后由 OpenGL 侧通过 `glGetUniformBlockIndex + glUniformBlockBinding` 显式把 `LightBlock` 绑定到 UBO binding point。
- **经验**：做多 API 或低版本 GL 兼容时，不能只看“现代 GLSL 写法是否更方便”，还要先对照目标版本上限。
- **优先检查项**：shader `#version`、目标 OpenGL 版本、是否使用了 4.2+ 才支持的 layout qualifier。
- **后续状态**：项目现已迁移到 OpenGL 4.5 Core，`LightBlock` 已恢复 `layout(std140, binding = 0)` 写法，并移除了 Program 侧的冗余 `glUniformBlockBinding`；这条记录保留用于解释当目标版本回退或多版本兼容时为什么会再次踩到。

### 11. 只升级 glad，不同步上下文版本和 GLSL 版本，会让项目处在半迁移状态

- **现象**：`glad` 头文件已经显示 `gl=4.5`，但运行时仍请求 OpenGL 4.1，上下文能力、shader 语法和文档约束彼此不一致，导致团队误判“某个特性能不能用”。
- **根因**：OpenGL 目标版本并不只由 `glad` 决定；它至少还受 GLFW 请求版本、shader `#version`、后端兼容分支、文档约定共同约束。
- **处理**：本次把 `glad`、GLFW 上下文请求、OpenGL shader 的 `#version`、`glClipControl` 路径和文档基线一起收口到 OpenGL 4.5 Core。
- **经验**：升级图形 API 版本时，要把“加载器、运行时上下文、shader 语言级别、文档”视为一个原子变更，而不是只换其中一环。
- **优先检查项**：`glad.h` 文件头、`GLFW_CONTEXT_VERSION_*`、shader `#version`、与版本绑定的兼容说明。

### 12. Sandbox 源码中的意外字符会阻塞全量构建

- **现象**：Release 全量构建在 `Source/Sandbox/Main.cpp:289` 失败，编译器报告 `w was not declared in this scope`。
- **根因**：`F2` 按键分支条件末尾存在意外字符 `w`，属于与本次渲染架构修改无关的现有源码问题。
- **处理**：本次未修改该无关源码；改为单独构建 `Renderer`、`RHIOpenGL`、`Engine` 目标验证架构修改。
- **优先检查项**：全量构建遇到突发未声明标识符时，先检查报错行末尾是否存在误输入字符，再判断是否与当前改动相关。

### 13. CPU 生成的 LUT 也会受到 OpenGL 纹理行序翻转影响

- **现象**：BRDF LUT 不是从图片文件读取，而是在 CPU 侧程序生成；但上传到 `OpenGLTexture` 时仍会走统一的 2D 纹理垂直翻转逻辑。
- **根因**：当前 `OpenGLTexture` 为了匹配普通图片资源的左上角原点约定，会对带初始数据的 2D 彩色纹理执行行序翻转。程序生成的数学 LUT 没有“图片原点”语义，如果不处理，shader 用 `roughness` 采样时会读到反向的 LUT 行。
- **处理**：生成 BRDF LUT 时反向写入 roughness 轴，使 OpenGL 上传翻转后仍与 shader 中 `vec2(NdotV, roughness)` 的采样约定一致。
- **优先检查项**：程序生成纹理、查找表、BRDF LUT、噪声纹理等是否真的应该走普通图片翻转路径。

### 14. 有 mipmap 数据不代表采样时真的用了 mipmap

- **现象**：几乎平行观察带高频纹理的表面时，纹理出现噪点和随相机移动剧烈抖动。
- **根因**：材质纹理创建时虽然生成了 mipmap，但绑定时使用的 OpenGL sampler object 只有 `GL_LINEAR` min filter。sampler object 会覆盖纹理对象自身的 min filter，因此实际采样没有走 `GL_LINEAR_MIPMAP_LINEAR`。同时缺少各向异性过滤会放大 grazing angle 走样。
- **处理**：`RHITextureFilter` 增加 `LinearMipmapLinear`，默认材质采样器与 Environment 采样器使用 mipmap 线性过滤，并在 OpenGL 支持 `GL_EXT_texture_filter_anisotropic` 时启用 8x 各向异性过滤。
- **优先检查项**：纹理是否生成 mip、sampler object 的 min filter、是否启用 anisotropy、GBuffer 是否误用了材质采样器。

### 15. 数学旋转约定不一致会让测试和相机语义同时失真

- **现象**：Release 下运行 `MathTest.exe` 时，`Quat Euler angles conversion` 和 `Transform LookAt GetForward` 两项失败。
- **根因**：`Quat::FromEuler(yaw, pitch, roll)` 使用 Y/X/Z 轴顺序构造，但原 `Quat::ToEulerAngles()` 公式并不是这一路径的严格逆变换；旧 `Transform::LookAt()` / `SetForward()` 使用 `forward x up` 构造右向量，在普通 Transform 约定“局部 +Z 为前向”下会生成反射基而不是有效旋转基。
- **处理**：已将 `ToEulerAngles()` 改为按 `yaw(Y) * pitch(X) * roll(Z)` 反解；当前 `Transform::LookAtRH()` 和 `SetForwardRH()` 使用 `up x forward` 构造右手正交基，并补充 MathTest 覆盖。矩阵 View/Projection API 也已改为 `LookAtRH/LH`、`PerspectiveRH/LH_ZO/NO` 这类显式命名，避免再次依赖默认手性。Release 下 `MathTest.exe` 已全部通过。
- **优先检查项**：欧拉角参数顺序、局部前向轴约定、`LookAt` 的 right/up/forward 正交基叉乘顺序、矩阵列主序和 `Transform::FromMatrix()` 的四元数提取。

### 16. `MemoryShutdown` 期间仍有 worker 使用分配器裸指针会触发访问冲突

- **现象**：Release 下运行 `SimpleTest.exe` 时，内存回归测试在输出一次 `DumpMemoryStats()` 后以 `0xC0000005` 访问冲突退出，没有到达 `all passed`。
- **根因**：`MemAlloc()` 内部只在函数调用期间持有 `std::shared_ptr<TlsfAllocator>`，返回给调用方的是裸指针。`MemoryShutdown()` 可在其它线程仍准备写入或释放该裸指针时销毁全局分配器并释放 TLSF pool，导致旧指针指向已释放的 OS 内存。`MemFree()` 也只查询当前全局 allocator，不能可靠知道指针来自哪个已被关闭的 allocator。
- **处理**：已按方案 A 修复：明确 `MemoryShutdown()` 只能在所有使用内存系统的 worker 停止、旧裸指针释放之后调用；`SimpleTest` 改为 `TestOrderedShutdownNoCrash()`，先 stop/join worker 再 shutdown。Release 下 `SimpleTest.exe` 已通过。
- **优先检查项**：是否有 worker 在线程停止前调用 `MemoryShutdown()`，是否还有旧裸指针未释放，是否有人把当前实现误用为支持并发 shutdown 的方案 B。

### 17. 将 ZO 投影矩阵按 NO 规则提取视锥 Near 平面会导致漏剔除

- **现象**：引擎使用 `PerspectiveRH_ZO()` 生成 `[0, 1]` 深度范围的投影矩阵，但视锥 Near 平面仍按 `row3 + row2` 提取。位于相机与近裁剪面之间的对象可能被错误视为在视锥内。
- **根因**：`row3 + row2` 对应 `[-1, 1]` 深度范围的裁剪条件 `z + w >= 0`；`ZO` 范围的 Near 条件是 `z >= 0`，应直接使用 `row2`。
- **处理**：将入口改为 `Frustum::FromViewProjectionRH_ZO()`，明确只接收 `ProjectionRH_ZO * ViewRH`，并按 ZO 规则提取 Near 平面；补充近裁剪面之前点的回归测试。
- **优先检查项**：投影 API 的 `RH/LH`、`ZO/NO` 后缀，VP 矩阵乘法顺序，以及 Frustum 平面提取公式是否匹配。

### 18. 局部旋转不能把世界轴与局部组合顺序混用

- **现象**：对象已经存在世界旋转时，调用 `RotateLocalX/Y/Z()` 得到的朝向不等于预期的局部轴旋转。
- **根因**：旧实现先通过 `GetRight/GetUp/GetForward()` 取得世界轴，却又按局部旋转的右乘顺序组合四元数，混合了两个坐标空间的规则。
- **处理**：`RotateLocal*()` 改为使用本地标准轴 `+X/+Y/+Z` 构造增量并右乘；`Rotate()` 和 `RotateLocal*()` 在组合后归一化四元数。
- **优先检查项**：旋转轴属于世界空间还是局部空间，四元数增量是左乘还是右乘，以及连续旋转后四元数长度是否仍接近 1。

### 19. 可执行文件能否直接启动不能依赖第三方库传递的链接参数

- **现象**：同一 `cmake-build-release/bin` 目录中的 `Sandbox.exe` 可以直接双击运行，而 `MathTest.exe`、`SimpleTest.exe` 以 `0xC0000135` 在进程启动阶段失败；三个文件没有显式可见的部署差异。
- **根因**：静态 Assimp 的 MinGW 配置向消费目标传递了 `-static-libgcc/-static-libstdc++/-lwinpthread`，使链接 Assimp 的 Sandbox 偶然静态归入运行库；只链接 Core 的测试目标仍依赖 `libgcc_s_seh-1.dll`、`libstdc++-6.dll` 与 `libwinpthread-1.dll`。可执行文件的部署性质因此错误地受第三方依赖图影响。
- **处理**：新增默认开启的项目级 `TE_STATIC_MINGW_RUNTIME` 选项，通过统一的可执行链接标志让所有 MinGW EXE 静态链接编译器运行库，不再依赖 Assimp 的间接行为。
- **经验**：运行库选择属于顶层产品构建策略，不能由某个第三方静态库的 CMake 脚本偶然决定；验证时应检查每个 EXE 的导入表，而不能只测试主程序。
- **优先检查项**：`TE_STATIC_MINGW_RUNTIME`、最终链接命令、`objdump -p <exe>` 的 DLL 导入表，以及新依赖是否提供静态库。

### 20. 反复覆盖同一 UBO 不能表达录制式 API 的每 Draw 数据快照

- **现象或风险**：OpenGL 中按 `UpdateData -> Draw -> UpdateData -> Draw` 覆盖同一 Object/Material UBO 可以显示不同对象；把同一上层流程直接映射到 Vulkan 后，多个已录制 Draw 可能全部读取最后一次写入的数据。
- **根因**：OpenGL Draw 立即消费当前状态，而 Vulkan CommandBuffer 记录的是未来要读取的 Buffer 范围，不会替普通主机写入保存历史版本。问题不是少调了一次 API，而是上层缺少“每次录制引用稳定范围”的语义。
- **处理**：新增按 frames-in-flight 分段的 transient Uniform ring、`DynamicUniformBuffer` 和 dynamic offset；Object、Material、Light、DeferredPass、Sky 每次上传获得独立范围，BindGroup 只持有稳定 ring buffer。
- **方案理由**：相比每对象创建 UBO，该方案减少小 Buffer/descriptor 数量；相比在更新时强制提交等待，不会把异步渲染串行化。OpenGL 可用 `glBindBufferRange` 实现同一契约。
- **优先检查项**：`AllocateTransientUniform()` 返回的 offset/range、后端 Uniform 对齐、`SetBindGroup()` dynamic offset 数量、当前 frame segment 是否仍被 GPU 使用，以及 Vulkan 原生资源是否通过 fence 延迟释放。

### 21. OpenGL 的隐式可见性会掩盖跨 Pass 资源状态缺失

- **现象或风险**：Deferred GBuffer 写完附件后立即在 Lighting Pass 采样，OpenGL 可以正常运行，但 Vulkan 无法知道 Image 应从 attachment layout 切到 shader-read layout，也无法生成正确的 stage/access 同步。
- **根因**：旧 RHI 只表达“开始一个 RenderPass”和“绑定一张纹理”，没有表达同一资源在 Pass 之间的用途变化；如果让 Vulkan 后端根据调用顺序猜测，资源状态会成为不可验证的后端隐式规则。
- **处理**：增加 `RHIResourceState`、`RHITextureBarrier` 与 `TransitionTexture()`；Deferred 在 GBuffer 前把颜色/深度切换到 `RenderTarget/DepthWrite`，写入后切换到 `ShaderResource`。OpenGL 保留状态跟踪并发出必要的 memory barrier。
- **方案理由**：状态由最了解 Pass 数据流的 Renderer 声明，后端只负责映射。当前先手写少量真实转换，未来 RenderGraph 可以消费同一状态契约。
- **优先检查项**：barrier 的 before 是否等于资源跟踪状态、附件是否同时声明 Attachment 与 ShaderResource usage、Vulkan 映射的 layout/stage/access 是否与实际读写阶段一致。

### 22. OpenGL 全局 binding 不能直接当作 Vulkan descriptor set 布局

- **现象或风险**：C++ 已把资源分为 group 0～4，但旧 GLSL 只有 `layout(binding = N)`；直接编译成 Vulkan SPIR-V 会把资源默认放进 set 0，与 PipelineLayout 不一致。手工复制一套 Vulkan Shader 又会迅速产生布局漂移。
- **根因**：OpenGL 使用全局 binding point，Vulkan 使用 `(set, binding)` 二元定位；旧 Shader 构建脚本也没有可执行的 `glslc` 查找、产物清单和失败策略。
- **处理**：新增共享 `TE_RESOURCE_BINDING/TE_UNIFORM_BINDING` 宏，OpenGL 展开为 binding，Vulkan 展开为 set + binding；OpenGL loader 支持相对路径 include；CMake 仅在 `TE_RHI_VULKAN=ON` 时要求 `glslc` 并生成 `VulkanShaders` 目标。
- **方案理由**：同一 GLSL 源成为跨后端 set/binding 的源级真相，同时 OpenGL-only 开发不承担 Vulkan SDK 依赖。自动反射仍可在该基础上增量引入。
- **优先检查项**：Shader 宏中的 group/binding、`RendererBindGroups`、`RendererBindings`、PipelineLayout 顺序、`VulkanShaders` 显式源清单，以及 Vulkan 配置时实际使用的 `glslc`。

### 23. Vulkan SDK 已安装不代表当前进程已经获得环境变量

- **现象**：SDK 已通过安装程序安装，系统 `vulkaninfo` 也能识别驱动，但当前终端中的 `VULKAN_SDK` 为空，`glslc` 不在 PATH，CMake 因而报告找不到工具链。
- **根因**：安装程序只更新后续创建进程继承的系统环境；已经打开的 Codex、CLion 或终端不会自动刷新。Vulkan Loader/驱动可由系统独立提供，所以 `vulkaninfo` 成功也不能证明 SDK Headers、Library 与 `glslc` 已被构建进程找到。
- **处理**：确认实际 SDK 根目录后，重启 IDE/终端，或在配置命令中显式设置 `VULKAN_SDK`、`Vulkan_ROOT` 并把 SDK `Bin` 加到当前进程 PATH。SDK 保持系统安装，不复制到项目目录。
- **优先检查项**：`VULKAN_SDK`、`Vulkan_ROOT`、`Get-Command glslc`、`glslc --version`、CMake 输出中的 `Vulkan_VERSION/Vulkan_LIBRARY`，以及 `vulkaninfo --summary` 显示的 Loader、Layer 和 GPU 驱动。

### 24. GLFW No-API 窗口不能调用依赖当前 OpenGL Context 的 VSync API

- **现象或风险**：Vulkan 使用 `GLFW_NO_API` 窗口时，沿用 OpenGL `SetVSync()` 实现会调用 `glfwSwapInterval()`；该函数要求当前线程存在 OpenGL/OpenGL ES Context，可能产生 GLFW 错误或污染启动诊断。
- **根因**：VSync 在 OpenGL 中通过 swap interval 控制，在 Vulkan 中属于 Swapchain present mode，两者不是同一个平台操作。窗口层此前把“保存用户偏好”和“立即修改 OpenGL Context”绑定在一起。
- **处理**：`GLFWWindow::SetVSync()` 始终保存偏好，但只在 `EWindowGraphicsAPI::OpenGL` 时调用 `glfwSwapInterval()`；Vulkan Device 在交换链创建/重建时读取偏好并选择 FIFO/MAILBOX/IMMEDIATE。
- **优先检查项**：窗口创建模式、是否存在当前 Context、`SetVSync` 调用位置、Swapchain present mode，以及切换偏好后是否标记交换链重建。

### 25. render-finished semaphore 不能仅按 frame slot 复用

- **现象**：双帧同步看似正常运行，但 Validation 报告 `vkQueueSubmit` 正在 signal 一个可能仍被 Swapchain 使用的 semaphore；最近 Acquire 的 image index 与 frame slot 轮转顺序并不一致。
- **根因**：submit fence 完成只证明 Graphics Queue 已执行完 signal，不能证明 Present Engine 已消费等待。若 render-finished semaphore 按 frame slot 分配，CPU 回到该 slot 时可能在前一次 Present 仍等待它的情况下再次 signal，违反二进制 semaphore 规则。
- **处理**：image-available semaphore 继续按 frame slot 管理；render-finished semaphore 改为每个 swapchain image 一个，并按 Acquire 返回的 image index 选择。相同图像再次 Acquire 可以作为上次 Present 已不再占用该 semaphore 的安全点。
- **方案理由**：该方案不要求额外 WSI 扩展，直接对齐交换链图像生命周期；frame slot 和 image index 的职责也保持清晰。
- **优先检查项**：`pSignalSemaphores` 的归属、Present 等待对象、交换链 image count、Acquire 顺序、是否错误地用 frame index 索引图像资源，以及是否考虑 `VK_KHR_swapchain_maintenance1` present fence 作为替代。

### 26. 最小化可在尺寸检查之后产生零 extent，并诱发主循环空转

- **现象**：BeginFrame 已检查传入尺寸非零，Validation 仍报告创建 `0×0` Swapchain 和零宽 Dynamic Rendering；同时最小化期间 FPS 统计达到百万级。
- **根因**：窗口最小化可以异步发生在入口检查与查询 surface capabilities 之间，最终 `currentExtent` 变为零。RHI 返回 `Skipped` 后，如果 Engine 不阻塞或节流，单线程主循环又会持续无提交空转。
- **处理**：Swapchain 创建在选出最终 `VkExtent2D` 后再次检查零尺寸，返回 `Skipped` 并保留待重建状态；Engine 对所有 `Skipped` 帧睡眠 16 ms，恢复后下一帧重建。
- **优先检查项**：最终 surface extent 而非仅缓存的窗口尺寸、`VK_ERROR_OUT_OF_DATE_KHR`、待重建标志、最小化恢复回调，以及 Skipped 帧是否有事件等待或有限节流。

### 27. Vulkan 的投影 Y 翻转与负 viewport height 只能选择一种

- **现象或风险**：阶段 A 只有固定清屏，无法观察几何方向；若同时在 `AdjustProjectionMatrix()` 翻转 Y，又使用负 viewport height，阶段 B 的网格会因双重适配而再次上下颠倒。
- **根因**：投影矩阵 Y 翻转和 `VK_KHR_maintenance1`/Vulkan 1.1 支持的负 viewport height 都能单独补偿 Vulkan NDC/Framebuffer Y 方向差异，两者叠加会互相抵消。
- **处理**：沿用项目既有多后端约定，在 `VulkanDevice::AdjustProjectionMatrix()` 翻转 Y；`VulkanCommandBuffer::SetViewport()` 使用正 height。阶段 B 用带明确朝向的模型验证方向，并在 Pipeline 中把 RHI CCW 映射为 Vulkan clockwise，以补偿投影翻转造成的 winding 变化。
- **优先检查项**：投影调整入口、viewport 的 `y/height`、FrontFace/CullMode、RT 采样翻转 trait，以及是否只有一个位置承担 Y 方向适配。

### 28. “能绘制”与“能运行完整场景”不能共用一个能力位

- **现象或风险**：基础 Buffer/Pipeline 完成后若直接打开完整场景门控，会立即触发 TextureCube、mip、IBL、MRT 等后续能力；继续关闭门控又只能清屏，无法验证真实 Draw。
- **根因**：单一 `bSupportsSceneRendering` 把后端 Bootstrap、基础绘制和完整应用场景压成了二值状态，阶段性实现没有可表达的中间能力。
- **处理**：保留 `bSupportsSceneRendering` 表示真实 Renderer 绘制能力，新增 `bSupportsFullSceneRendering` 控制 Sandbox 场景与每帧回调。Vulkan 阶段 B 使用 Renderer 内部静态网格验证路径，OpenGL 保持完整场景。
- **方案理由**：能力门控描述功能，不把 Vulkan 类型判断散落到 Engine/Renderer，也不会为了通过初始化而返回伪资源。
- **优先检查项**：Engine 场景回调条件、SceneRenderer 路由、后端 traits、验证路径是否泄漏成应用公共 API，以及日志是否准确说明当前模式。

### 29. Vulkan 投影 Y 翻转还会改变正面缠绕

- **现象或风险**：纹理与几何上下方向看似正确，但启用 BackFace Culling 后整个模型消失，关闭裁剪又能显示。
- **根因**：在投影矩阵中把 Y 乘以 -1 是一次镜像变换，会把屏幕空间三角形的 clockwise/counter-clockwise 方向对调；只处理 viewport 方向而不处理 FrontFace 会错误剔除。
- **处理**：项目保持正 viewport height，并在 Vulkan Pipeline 创建时把 RHI `CounterClockwise` 映射为 `VK_FRONT_FACE_CLOCKWISE`，反向亦然。
- **优先检查项**：`AdjustProjectionMatrix`、native frontFace、CullMode、顶点索引顺序，以及是否同时使用了负 viewport height。

### 30. 原生句柄已创建不代表 RHI 资源构造完整成功

- **现象或风险**：Buffer/Image 原生对象已创建，但映射、初始上传或 Descriptor entry 校验失败；若 `IsValid()` 只检查句柄，工厂仍返回半初始化资源。DescriptorSet 分配后直接把句柄清零还会泄漏池内配额。
- **根因**：Vulkan 资源创建是多步骤事务，任一步失败都应使包装对象无效并回滚已取得的对象；只以首个句柄作为成功标志忽略了后续步骤。
- **处理**：Buffer/Texture 使用仅在构造全流程结束时设置的显式 valid 状态；工厂按该状态决定是否返回。Descriptor entry 失败时先 `vkFreeDescriptorSets`，再清空句柄。
- **优先检查项**：内存绑定、映射、初始上传、ImageView、layout transition、Descriptor entry 完整性、失败路径清理顺序，以及 pool 配额是否被静默耗尽。

## 二、当前仍需警惕的潜在问题

### 1. GBuffer 目前以正确性优先，带宽成本偏高

- **风险**：正式 Deferred 光照已经依赖深度重建，但 GBuffer 仍额外写入并保存 `WorldPosition` 供调试对照，尚未回收 RT 显存与写带宽。
- **建议**：正式 Deferred 光照已切换到重建位置，Position RT 继续作为安全对照。使用 `F8` 校准：黑色 `<1 μm`，蓝色 `[1,10) μm`，青色 `[10,100) μm`，绿色 `[0.1,1) mm`，黄色 `[1,10) mm`，红色 `≥1 cm`；若光照异常，先比较 F6/F8，再决定回退或继续移除 RT。

### 2. Deferred 路径仍不支持半透明

- **风险**：继续向真实场景扩展时，透明对象无法直接纳入当前 Deferred GBuffer。
- **建议**：保持“Opaque Deferred + Translucent Forward”这条演进路线。

### 3. 颜色空间与 sRGB 仍需统一策略

- **风险**：Albedo、RT 写入、framebuffer sRGB 状态若不统一，会出现整体偏暗/偏亮；Normal / Depth 若误设为 sRGB 会直接损坏数据。
- **建议**：继续把“哪些通道允许 sRGB、哪些绝不允许”写死到格式策略里。

### 4. RenderTarget 仍只覆盖单 mip-level / 非 layered 用法

- **风险**：后续若接 Cubemap、3D Texture Slice、Layered RT，会直接碰到当前 FBO 封装的边界。
- **建议**：在引入相关功能前先扩展 RT 抽象，不要到具体特性落地时临时补洞。

### 5. 资源 binding 若同时写在 shader 和 C++ layout 里，布局漂移仍然可能发生

- **风险**：当前主渲染路径已经引入 BindGroupLayout / PipelineLayout，shader 也通过公共宏声明 group/set + binding，但 Shader 宏参数与 C++ `RendererBindGroups/RendererBindings` 仍是两份声明。
- **根因**：项目目前还没有 shader 反射或代码生成链路，运行时代码无法自动从 shader 布局导出 binding 信息。
- **处理**：现阶段通过公共 GLSL 宏统一 OpenGL/Vulkan Shader 声明，通过 `RendererBindGroups` / `RendererBindings` 管理 C++ 约定，并用 BindGroupLayout / PipelineLayout 做创建期结构校验。
- **建议**：后续建立 SPIR-V / GLSL 反射或资源布局生成流程，让 binding 描述有单一真相源。
- **优先检查项**：shader `TE_RESOURCE_BINDING/TE_UNIFORM_BINDING` 参数、`RendererBindings`、`RendererBindGroups`、BindGroupLayout、PipelineLayout。

## 三、阅读与排查建议

遇到新问题时，建议按下面顺序排查：

1. 先判断是环境问题、运行时状态问题，还是源码逻辑问题。
2. 若是图形输出异常，先确认“坐标约定、采样器、颜色空间、深度范围”四类基础约定是否一致。
3. 若是 Deferred 问题，优先确认 GBuffer 输入是否真的是你以为写进去的值。
4. 若是 Forward / Deferred 结果差异异常，先比较两边 shader 的数学公式，而不是先猜 OpenGL 状态。
5. 问题修掉后，回到本文档补记录，避免同类问题再次依赖记忆。
