# 图形与开发踩坑记录

本文档用于长期沉淀本仓库在图形学实现、运行时架构、工具链使用过程中已经遇到过的 bug、误判、实现陷阱与仍需警惕的潜在问题。

它和 [`临时架构问题清单`](./临时架构问题清单.md) 的区别是：
- `临时架构问题清单` 更偏“尚未整改的结构性问题”
- 本文档更偏“已经踩过或明确识别过的坑”，包括已修复问题、定位经验、以及值得反复提醒的工程误区

目标：
- 给后续开发提供可复用的排障经验
- 把“为什么会错”记录下来，而不仅是“后来怎么改”
- 帮助学习图形学与引擎开发中的真实实现细节

## 维护规则

- 每次出现明确 bug，修复时都应在本文档补一条记录。
- 每次识别出虽然暂未触发、但后续高概率踩中的潜在问题，也应补一条记录。
- 记录必须至少说明：
  - 现象或风险是什么
  - 根因是什么
  - 最终如何处理，或为什么暂时不处理
  - 后续再遇到时应优先检查什么
- 若某条记录后来被新的实现覆盖，不删除旧记录，而是在原条目下补“后续状态”。

## 一、已修复或已明确定位的坑

### 1. Deferred RT 采样 Y 翻转误判

- **现象**：切到 Deferred 后整张画面上下颠倒，鼠标抬头/低头视觉上也反了，光照方向看起来不对。
- **根因**：Lighting Pass 里对 RT 采样又做了一次 `Y` 翻转，但当前 OpenGL 全屏三角形路径的 UV 已经和 FBO 纹理坐标匹配，不需要再翻。
- **处理**：将 `RHIBackendTraits::bRTSampleRequiresFlipY` 在当前 OpenGL 路径上设为 `false`，并保留该 trait 作为后端约定显式入口。
- **经验**：RT 采样坐标是否需要翻转，必须作为“后端约定”显式管理，不能凭印象在 shader 里硬编码。
- **优先检查项**：`RHIBackendTraits`、全屏三角形 UV 生成方式、FBO 纹理原点约定。

### 2. Deferred Lighting 误用默认线性采样器

- **现象**：Deferred 路径下，斜着看某个面时异常变亮，正着看又变暗。
- **根因**：Lighting Pass 复用了默认 `Linear` sampler 去采样 GBuffer。BaseColor 线性过滤通常可以接受，但 Normal 和尤其是 Depth 不能被线性插值，否则会破坏延迟渲染输入。
- **处理**：新增专用 `GBuffer_Nearest_Sampler`，统一对 GBuffer 与 Depth 使用 `Nearest + ClampToEdge`。
- **经验**：GBuffer 不是普通颜色贴图。法线、深度、材质参数、ID 这类“结构化数据”默认都应按离散值处理，不能先假定线性过滤安全。
- **优先检查项**：Lighting Pass 的 sampler 绑定、法线编码、Depth 采样方式。

### 3. Deferred 依赖深度重建世界坐标导致点光视角相关误差

- **现象**：即使修正了 GBuffer 采样器，Deferred 下同一表面的点光亮度仍随观察角度明显变化。
- **根因**：当前 OpenGL 深度约定、投影矩阵适配和 `inverse(ViewProjection)` 重建链路之间仍存在不一致，导致 `Depth -> WorldPosition` 反推结果不稳定。
- **处理**：GBuffer 新增 `WorldPosition` 通道，Lighting Pass 直接读取世界坐标，不再依赖深度重建。
- **代价**：额外增加一张 `RGBA32_Float` RT，带宽和显存成本上升。
- **经验**：当多后端深度约定尚未完全校准时，位置重建很容易成为隐性误差源。学习阶段先用显式 Position RT 保证正确性是合理取舍。
- **优先检查项**：NDC 深度范围、`AdjustProjectionMatrix()`、深度纹理采样值、重建公式与 API 约定是否一致。
- **后续状态**：未来若要移除 Position RT，必须单独验证深度重建链路，而不是直接回退。

### 4. Forward / Deferred 切换后视觉上可能完全一致

- **现象**：按 `F1` / `F2` 切换时，画面看起来几乎没有区别。
- **根因**：两条路径当前使用的是同一套基础光照公式，只是执行位置不同：Forward 在 BasePass 中算，Deferred 在 Lighting Pass 中算。
- **处理**：保留该行为，不视为 bug。
- **经验**：渲染路径切换不等于着色模型切换。若要验证路径切换，应增加调试视图、路径提示或统计信息，而不是用“肉眼看起来应该不同”判断。
- **优先检查项**：日志是否打印路径切换、Shader 公式是否本来就等价、是否存在调试视图。

### 5. Sandbox 可执行文件占用导致 Windows 链接失败

- **现象**：构建走到最后链接 `bin/Sandbox.exe` 时失败，提示 `Permission denied` 或拒绝访问。
- **根因**：Windows 下正在运行的可执行文件会锁定目标输出，链接器无法覆盖。
- **处理**：关闭正在运行的 `Sandbox.exe` 后重新构建。
- **经验**：这种失败首先归类为“运行中的产物占用”，不要一上来怀疑源码或链接参数。
- **优先检查项**：是否仍开着 Sandbox、任务管理器里的进程状态、输出文件占用情况。

### 6. CMake / g++ 调用失败不一定是源码问题

- **现象**：`cmake --build` 或 `g++.exe` 无法正常运行。
- **根因**：CLion 自带 MinGW 工具链的 `PATH` 未注入当前进程，或者当前 PowerShell 环境不满足该调用方式。
- **处理**：优先补齐 `C:\Program Files\JetBrains\CLion 2025.3.3\bin\mingw\bin` 到 `PATH`，使用仓库约定的 CLion CMake 可执行文件，再判断是否为代码问题。
- **经验**：工具链环境未校准前，不要把“无法构建”归因到源码。
- **优先检查项**：`PATH`、CMake 可执行文件路径、构建目录、当前 shell 权限和环境。

### 7. 法线不能直接用 `mat3(model)` 变换

- **现象**：在存在非等比缩放的模型上，光照方向会悄悄出错，表现为亮面和暗面的分布不符合几何直觉。
- **根因**：法线不是普通位置向量，不能直接乘 `mat3(u_Model)`；正确做法是使用模型矩阵左上角 `3x3` 的逆转置。
- **处理**：Forward / Deferred 顶点 shader 统一改为使用 `u_NormalMatrix`，CPU 侧用 `WorldMatrix.GetNormalMatrix()` 上传。
- **经验**：只要涉及光照，就不要默认“位置矩阵也能直接变法线”。等比缩放时看起来没问题，不代表公式是对的。
- **优先检查项**：shader 里的 normal transform、CPU 上传的 normal matrix、是否存在非等比缩放。

### 8. 路径切换需要独立的调试反馈

- **现象**：Forward / Deferred 使用同一套光照公式时，切换后肉眼可能几乎看不出区别，容易误判成切换没生效。
- **根因**：渲染路径差异在于 Pass 组织方式，不一定体现在最终颜色公式上。
- **处理**：为 Sandbox 增加窗口标题状态显示和 Deferred GBuffer 调试视图快捷键（`F3-F7`）。
- **经验**：做渲染系统时，验证能力本身也是功能的一部分；没有 debug view，很容易把“看起来没变化”误认为 bug。
- **优先检查项**：当前路径状态、调试视图模式、日志是否有切换记录。

### 9. OpenGL 4.1 不能直接在 UBO 上写 `layout(binding = N)`

- **现象**：shader 编译失败，错误类似：`layout qualifier 'binding' requires "#version 420" or later`。
- **根因**：问题出现时项目目标还是 OpenGL 4.1 Core；`layout(binding = N)` 这类 uniform block binding 语法需要 GLSL 4.20 或 `GL_ARB_shading_language_420pack`。
- **处理**：shader 保留 `layout(std140) uniform LightBlock`，不在 GLSL 中写 binding；Program 链接后由 OpenGL 侧通过 `glGetUniformBlockIndex + glUniformBlockBinding` 显式把 `LightBlock` 绑定到 UBO binding point。
- **经验**：做多 API 或低版本 GL 兼容时，不能只看“现代 GLSL 写法是否更方便”，还要先对照目标版本上限。
- **优先检查项**：shader `#version`、目标 OpenGL 版本、是否使用了 4.2+ 才支持的 layout qualifier。
- **后续状态**：项目现已迁移到 OpenGL 4.5 Core，`LightBlock` 已恢复 `layout(std140, binding = 0)` 写法，并移除了 Program 侧的冗余 `glUniformBlockBinding`；这条记录保留用于解释当目标版本回退或多版本兼容时为什么会再次踩到。

### 10. 只升级 glad，不同步上下文版本和 GLSL 版本，会让项目处在半迁移状态

- **现象**：`glad` 头文件已经显示 `gl=4.5`，但运行时仍请求 OpenGL 4.1，上下文能力、shader 语法和文档约束彼此不一致，导致团队误判“某个特性能不能用”。
- **根因**：OpenGL 目标版本并不只由 `glad` 决定；它至少还受 GLFW 请求版本、shader `#version`、后端兼容分支、文档约定共同约束。
- **处理**：本次把 `glad`、GLFW 上下文请求、OpenGL shader 的 `#version`、`glClipControl` 路径和文档基线一起收口到 OpenGL 4.5 Core。
- **经验**：升级图形 API 版本时，要把“加载器、运行时上下文、shader 语言级别、文档”视为一个原子变更，而不是只换其中一环。
- **优先检查项**：`glad.h` 文件头、`GLFW_CONTEXT_VERSION_*`、shader `#version`、与版本绑定的兼容说明。

### 11. Sandbox 源码中的意外字符会阻塞全量构建

- **现象**：Release 全量构建在 `Source/Sandbox/Main.cpp:289` 失败，编译器报告 `w was not declared in this scope`。
- **根因**：`F2` 按键分支条件末尾存在意外字符 `w`，属于与本次渲染架构修改无关的现有源码问题。
- **处理**：本次未修改该无关源码；改为单独构建 `Renderer`、`RHIOpenGL`、`Engine` 目标验证架构修改。
- **优先检查项**：全量构建遇到突发未声明标识符时，先检查报错行末尾是否存在误输入字符，再判断是否与当前改动相关。

### 12. CPU 生成的 LUT 也会受到 OpenGL 纹理行序翻转影响

- **现象**：BRDF LUT 不是从图片文件读取，而是在 CPU 侧程序生成；但上传到 `OpenGLTexture` 时仍会走统一的 2D 纹理垂直翻转逻辑。
- **根因**：当前 `OpenGLTexture` 为了匹配普通图片资源的左上角原点约定，会对带初始数据的 2D 彩色纹理执行行序翻转。程序生成的数学 LUT 没有“图片原点”语义，如果不处理，shader 用 `roughness` 采样时会读到反向的 LUT 行。
- **处理**：生成 BRDF LUT 时反向写入 roughness 轴，使 OpenGL 上传翻转后仍与 shader 中 `vec2(NdotV, roughness)` 的采样约定一致。
- **优先检查项**：程序生成纹理、查找表、BRDF LUT、噪声纹理等是否真的应该走普通图片翻转路径。

### 13. 有 mipmap 数据不代表采样时真的用了 mipmap

- **现象**：几乎平行观察带高频纹理的表面时，纹理出现噪点和随相机移动剧烈抖动。
- **根因**：材质纹理创建时虽然生成了 mipmap，但绑定时使用的 OpenGL sampler object 只有 `GL_LINEAR` min filter。sampler object 会覆盖纹理对象自身的 min filter，因此实际采样没有走 `GL_LINEAR_MIPMAP_LINEAR`。同时缺少各向异性过滤会放大 grazing angle 走样。
- **处理**：`RHITextureFilter` 增加 `LinearMipmapLinear`，默认材质采样器与 Environment 采样器使用 mipmap 线性过滤，并在 OpenGL 支持 `GL_EXT_texture_filter_anisotropic` 时启用 8x 各向异性过滤。
- **优先检查项**：纹理是否生成 mip、sampler object 的 min filter、是否启用 anisotropy、GBuffer 是否误用了材质采样器。

### 14. 数学旋转约定不一致会让测试和相机语义同时失真

- **现象**：Release 下运行 `MathTest.exe` 时，`Quat Euler angles conversion` 和 `Transform LookAt GetForward` 两项失败。
- **根因**：`Quat::FromEuler(yaw, pitch, roll)` 使用 Y/X/Z 轴顺序构造，但原 `Quat::ToEulerAngles()` 公式并不是这一路径的严格逆变换；旧 `Transform::LookAt()` / `SetForward()` 使用 `forward x up` 构造右向量，在普通 Transform 约定“局部 +Z 为前向”下会生成反射基而不是有效旋转基。
- **处理**：已将 `ToEulerAngles()` 改为按 `yaw(Y) * pitch(X) * roll(Z)` 反解；当前 `Transform::LookAtRH()` 和 `SetForwardRH()` 使用 `up x forward` 构造右手正交基，并补充 MathTest 覆盖。矩阵 View/Projection API 也已改为 `LookAtRH/LH`、`PerspectiveRH/LH_ZO/NO` 这类显式命名，避免再次依赖默认手性。Release 下 `MathTest.exe` 已全部通过。
- **优先检查项**：欧拉角参数顺序、局部前向轴约定、`LookAt` 的 right/up/forward 正交基叉乘顺序、矩阵列主序和 `Transform::FromMatrix()` 的四元数提取。

### 15. `MemoryShutdown` 期间仍有 worker 使用分配器裸指针会触发访问冲突

- **现象**：Release 下运行 `SimpleTest.exe` 时，内存回归测试在输出一次 `DumpMemoryStats()` 后以 `0xC0000005` 访问冲突退出，没有到达 `all passed`。
- **根因**：`MemAlloc()` 内部只在函数调用期间持有 `std::shared_ptr<TlsfAllocator>`，返回给调用方的是裸指针。`MemoryShutdown()` 可在其它线程仍准备写入或释放该裸指针时销毁全局分配器并释放 TLSF pool，导致旧指针指向已释放的 OS 内存。`MemFree()` 也只查询当前全局 allocator，不能可靠知道指针来自哪个已被关闭的 allocator。
- **处理**：已按方案 A 修复：明确 `MemoryShutdown()` 只能在所有使用内存系统的 worker 停止、旧裸指针释放之后调用；`SimpleTest` 改为 `TestOrderedShutdownNoCrash()`，先 stop/join worker 再 shutdown。Release 下 `SimpleTest.exe` 已通过。
- **优先检查项**：是否有 worker 在线程停止前调用 `MemoryShutdown()`，是否还有旧裸指针未释放，是否有人把当前实现误用为支持并发 shutdown 的方案 B。

### 16. 将 ZO 投影矩阵按 NO 规则提取视锥 Near 平面会导致漏剔除

- **现象**：引擎使用 `PerspectiveRH_ZO()` 生成 `[0, 1]` 深度范围的投影矩阵，但视锥 Near 平面仍按 `row3 + row2` 提取。位于相机与近裁剪面之间的对象可能被错误视为在视锥内。
- **根因**：`row3 + row2` 对应 `[-1, 1]` 深度范围的裁剪条件 `z + w >= 0`；`ZO` 范围的 Near 条件是 `z >= 0`，应直接使用 `row2`。
- **处理**：将入口改为 `Frustum::FromViewProjectionRH_ZO()`，明确只接收 `ProjectionRH_ZO * ViewRH`，并按 ZO 规则提取 Near 平面；补充近裁剪面之前点的回归测试。
- **优先检查项**：投影 API 的 `RH/LH`、`ZO/NO` 后缀，VP 矩阵乘法顺序，以及 Frustum 平面提取公式是否匹配。

### 17. 局部旋转不能把世界轴与局部组合顺序混用

- **现象**：对象已经存在世界旋转时，调用 `RotateLocalX/Y/Z()` 得到的朝向不等于预期的局部轴旋转。
- **根因**：旧实现先通过 `GetRight/GetUp/GetForward()` 取得世界轴，却又按局部旋转的右乘顺序组合四元数，混合了两个坐标空间的规则。
- **处理**：`RotateLocal*()` 改为使用本地标准轴 `+X/+Y/+Z` 构造增量并右乘；`Rotate()` 和 `RotateLocal*()` 在组合后归一化四元数。
- **优先检查项**：旋转轴属于世界空间还是局部空间，四元数增量是左乘还是右乘，以及连续旋转后四元数长度是否仍接近 1。

## 二、当前仍需警惕的潜在问题

### 1. `RHICommandBuffer` 仍无显式资源屏障接口

- **风险**：OpenGL 依赖隐式同步还能工作，但 Vulkan / D3D12 后端接入前这会直接成为功能阻塞项。
- **建议**：补 `TransitionResource` 或等价接口，并先确定跨后端语义。

### 2. Shader 跨后端编译管线尚未建立

- **风险**：目前仍以手工组织 shader 资产为主，多后端越多，分发和反射越容易失控。
- **建议**：建立 GLSL -> SPIR-V -> 反射/交叉编译的正式流程。

### 3. GBuffer 目前以正确性优先，带宽成本偏高

- **风险**：当前 Deferred 额外保存 `WorldPosition`，而非依赖深度重建，正确性更稳，但 RT 成本更高。
- **建议**：后续若要优化带宽，应先单独重做深度重建校准，再评估是否移除 Position RT。

### 4. Deferred 路径仍不支持半透明

- **风险**：继续向真实场景扩展时，透明对象无法直接纳入当前 Deferred GBuffer。
- **建议**：保持“Opaque Deferred + Translucent Forward”这条演进路线。

### 5. 颜色空间与 sRGB 仍需统一策略

- **风险**：Albedo、RT 写入、framebuffer sRGB 状态若不统一，会出现整体偏暗/偏亮；Normal / Depth 若误设为 sRGB 会直接损坏数据。
- **建议**：继续把“哪些通道允许 sRGB、哪些绝不允许”写死到格式策略里。

### 6. RenderTarget 仍只覆盖单 mip-level / 非 layered 用法

- **风险**：后续若接 Cubemap、3D Texture Slice、Layered RT，会直接碰到当前 FBO 封装的边界。
- **建议**：在引入相关功能前先扩展 RT 抽象，不要到具体特性落地时临时补洞。

### 7. 资源 binding 若同时写在 shader 和 C++ layout 里，布局漂移仍然可能发生

- **风险**：当前主渲染路径已经引入 BindGroupLayout / PipelineLayout，但 shader 里的 `layout(binding = N)` 与 C++ 里的 `RendererBindings` 仍是两份声明。
- **根因**：项目目前还没有 shader 反射或代码生成链路，运行时代码无法自动从 shader 布局导出 binding 信息。
- **处理**：现阶段先通过 `RendererBindGroups` / `RendererBindings` 分离 group index 与 binding slot，并用 BindGroupLayout / PipelineLayout 做创建期结构校验。
- **建议**：后续建立 SPIR-V / GLSL 反射或资源布局生成流程，让 binding 描述有单一真相源。
- **优先检查项**：shader `layout(binding = N)`、`RendererBindings`、`RendererBindGroups`、BindGroupLayout、PipelineLayout。

## 三、阅读与排查建议

遇到新问题时，建议按下面顺序排查：

1. 先判断是环境问题、运行时状态问题，还是源码逻辑问题。
2. 若是图形输出异常，先确认“坐标约定、采样器、颜色空间、深度范围”四类基础约定是否一致。
3. 若是 Deferred 问题，优先确认 GBuffer 输入是否真的是你以为写进去的值。
4. 若是 Forward / Deferred 结果差异异常，先比较两边 shader 的数学公式，而不是先猜 OpenGL 状态。
5. 问题修掉后，回到本文档补记录，避免同类问题再次依赖记忆。
