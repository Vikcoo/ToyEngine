# 构建与运行

## 环境要求

- `CMake 3.18+`
- 支持项目所需标准的 C++ 编译器
- 选择 OpenGL 时需要可用的 OpenGL 4.5 运行环境
- 选择 Vulkan 时需要 Vulkan 1.3 Loader/驱动、Vulkan SDK Headers/Loader Library、`glslc`，开发验证默认还需要 `VK_LAYER_KHRONOS_validation`
- 当前工程配置所依赖的第三方库与工具链

只构建 OpenGL 后端时不需要 Vulkan SDK。Vulkan SDK 应安装到系统或开发机统一目录，不需要复制到项目或提交仓库；CMake 通过 `VULKAN_SDK`、`Vulkan_ROOT` 和 `PATH` 定位它。安装程序完成后，已经打开的终端/IDE 可能仍没有刷新环境变量，此时应重启进程或在本次配置命令中显式设置路径。

当前 CMake 强制 OpenGL/Vulkan 二选一。`TE_RHI_D3D12` 仍是保留选项，启用会明确配置失败，不能被误认为可运行后端。Vulkan 开发配置默认开启 `TE_VULKAN_VALIDATION`；若验证层缺失，Device 初始化会失败并给出明确日志。

## 配置

常见本地配置方式：

```powershell
cmake -S . -B cmake-build-release -DCMAKE_BUILD_TYPE=Release
```

当前默认约定：
- 常规开发验证、AI 构建检查、构建耗时分析默认使用 `Release`
- 默认构建目录为 `cmake-build-release`
- 只有在需要断点调试、符号排查或验证 Debug 专属问题时，才显式切换到 `cmake-build-debug`
- CLion MinGW 构建默认启用 `TE_STATIC_MINGW_RUNTIME=ON`，所有可执行目标静态链接 GCC、C++ 标准库与 winpthreads 运行库

当前仓库的构建入口仍然是标准 CMake 配置；`CMake/` 目录目前只包含平台检测、基础选项和 Shader 相关辅助脚本，还没有把 Unity Build、预编译头、编译缓存等“构建加速”能力正式接入到各个目标。

### Vulkan 阶段 B 配置与 Shader 编译

由于 OpenGL 默认开启，Vulkan 配置必须显式关闭 OpenGL。以下示例使用独立构建目录，避免覆盖日常 OpenGL cache：

```powershell
$env:VULKAN_SDK = 'C:\VulkanSDK\1.4.350.0'
$env:PATH = "$env:VULKAN_SDK\Bin;$env:PATH"
cmake -S . -B cmake-build-vulkan -G Ninja `
  -DCMAKE_BUILD_TYPE=Release `
  -DTE_RHI_OPENGL=OFF `
  -DTE_RHI_VULKAN=ON `
  -DTE_RHI_D3D12=OFF `
  -DTE_VULKAN_VALIDATION=ON `
  -DVulkan_ROOT="$env:VULKAN_SDK"
cmake --build cmake-build-vulkan -j 8
```

示例版本只是本次验收路径，不是项目锁定版本；应替换为实际安装目录。CMake 会通过 `find_package(Vulkan 1.3 REQUIRED)` 验证 Headers/Loader，并对根清单中的 9 个 Shader 定义 `TE_RHI_VULKAN=1`，生成到构建目录的 `CompiledShaders/Vulkan/`。其中两个 `Vulkan/stage_b_validation.*` 是阶段 B 静态网格验证资产，其余为后续完整路径准备的主 Shader。缺少 `glslc` 会在配置阶段明确失败；OpenGL-only 配置不生成该目标。本次阶段 B 已用 SDK 1.4.350、Vulkan Loader 1.4.350 和 RTX 5070 Ti 驱动完成实际编译与 Validation 运行验收。

Vulkan Sandbox 当前进入阶段 B 验证模式：创建 No-API 窗口后，通过 Renderer 内部路径持续绘制一个带非对称 sRGB 纹理的旋转 indexed cube，覆盖 GPU-only Buffer/Image 上传、Descriptor、dynamic Uniform、Graphics Pipeline、D32 深度与 Reversed-Z。它仍不会执行 Sandbox 的完整场景初始化与每帧回调，也不代表 Forward/Deferred/PBR 已达到 OpenGL 功能对等。

当前状态说明：
- `TE_ENABLE_UNITY_BUILD`、`TE_ENABLE_PCH`、`TE_UNITY_BUILD_BATCH_SIZE` 这些缓存项可能出现在已有构建目录中，但仓库内现有 `CMakeLists.txt` 尚未消费这些选项
- 因此是否传入这些选项，当前默认工程都不会自动为 Runtime 模块生成 Unity 源文件或预编译头
- 如果后续将这些能力正式接入，必须在同一变更中同步更新本页

### MinGW 静态运行库策略

`CMake/EngineOptions.cmake` 集中定义 `TE_STATIC_MINGW_RUNTIME`，默认值为 `ON`。启用后，CMake 对所有当前及未来的可执行目标统一追加 `-static -static-libgcc -static-libstdc++`，使 `Sandbox.exe`、`MathTest.exe`、`SimpleTest.exe` 等产物不再依赖以下 MinGW 动态运行库：

- `libgcc_s_seh-1.dll`
- `libstdc++-6.dll`
- `libwinpthread-1.dll`

这里的“静态”仅指可由 MinGW 静态归入程序的编译器运行库和当前静态第三方依赖；Windows 系统 DLL、图形驱动和操作系统 API 仍然是正常的运行前提，不追求生成完全不导入系统 DLL 的二进制。

该策略必须由项目自身显式持有，不能依赖 Assimp 等第三方目标偶然传递静态链接参数。若需要专项验证动态运行库部署，可在配置时显式关闭：

```powershell
cmake -S . -B cmake-build-release -DCMAKE_BUILD_TYPE=Release -DTE_STATIC_MINGW_RUNTIME=OFF
```

启用全静态选择后，新引入的第三方库必须提供兼容的静态库；如果只有 DLL/import library，应重新评估该依赖的部署方式，而不是在单个可执行目标中偷偷覆盖全局约定。

## 运行

运行构建目录中生成的 Sandbox 可执行文件。实际路径取决于 CMake 生成器和构建配置。

启动时的当前行为：
- 引擎先创建窗口并初始化所选 RHI
- OpenGL 完整场景模式会尝试从 `Content/Models/orientation_cube.obj` 加载示例模型；导入失败时创建内置立方体，并创建飞行相机
- Vulkan 阶段 B 会延后应用场景回调，改由 Renderer 内部静态网格验证路径绘制一个旋转 cube
- 调试日志每 `0.5` 秒输出平均 FPS、当前渲染相机世界坐标 `CameraWS: (X, Y, Z)` 以及 DrawCall / Pipeline / VBO / IBO 绑定统计

## 日志文件

运行 Sandbox 或调用 `TE::Log::Init()` 的测试程序后，日志会同时输出到控制台和引擎根目录下的 `Saved/Logs/ToyEngine.log`。日志目录会自动创建且不纳入 Git 管理；单个文件达到 `5 MiB` 后自动滚动，并最多保留 3 个历史文件。

程序正常退出时必须在所有日志生产线程停止、最后一条日志写入之后调用 `TE::Log::Shutdown()`，以刷新异步队列。若进程被强制终止，最近 3 秒内尚未定期刷新的普通日志仍可能丢失。

## 测试

`Tests/CMakeLists.txt` 会把每个 `Tests/*.cpp` 单独编译成一个可执行文件。普通测试链接 `Core`；`RHITransientAllocatorTest` 链接 `RHI`，但不创建图形上下文。

当前含义：
- 测试较轻量，偏模块级
- 还不是完整的引擎集成测试
- `RHITransientAllocatorTest` 覆盖 Uniform ring 的初始化约束、对齐、溢出、帧段隔离和帧索引回绕
- MinGW 下测试可执行文件遵循与 Sandbox 相同的静态运行库策略，可以从 `cmake-build-release/bin` 直接启动，无需额外配置工具链 `PATH`
