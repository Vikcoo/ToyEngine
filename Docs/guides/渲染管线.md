# 渲染管线

本文档描述当前代码里真实存在的渲染路径，而不是理想化设计。

## 当前渲染模型

当前渲染器是一个单线程、单 Pass 的简化路径，核心阶段如下：
- 游戏侧 `Primitive` 通过 `IRenderSceneBridge` 请求创建渲染侧 `SceneProxy`
- `FScene` 保存当前活跃的渲染侧 Primitive 列表与视图数据
- `SceneRenderer` 从各个 Proxy 收集 `FMeshDrawCommand`
- 对 DrawCommand 做排序，减少状态切换
- 通过 `RHICommandBuffer` 提交命令
- 最终由 `OpenGLRHI` 执行实际图形 API 调用

## 关键类型

### `FScene`
职责：
- 保存当前活跃的 Primitive Proxy
- 保存当前帧的 `FViewInfo`

### `FPrimitiveSceneProxy`
职责：
- 作为游戏侧 `PrimitiveComponent` 的渲染镜像
- 对外提供绘制命令
- 持有渲染侧所需的世界变换副本等状态

### `FStaticMeshSceneProxy`
职责：
- 把 `TStaticMesh` 中的 Section 数据转换为 GPU 缓冲
- 为自己代表的各个 Section 生成一条或多条 DrawCommand

### `FMeshDrawCommand`
职责：
- 打包一次绘制提交所需的状态
- 把命令收集阶段和命令提交阶段解耦

### `SceneRenderer`
职责：
- 从所有活跃 Proxy 收集绘制命令
- 按 Pipeline 和 Buffer 状态排序，减少冗余绑定
- 提交命令到 `RHICommandBuffer`
- 维护轻量级帧统计

## 游戏侧到渲染侧的桥接

桥接过程始于 `TPrimitiveComponent`：
- 它通过 `BuildRenderCreateInfo` 生成渲染创建描述
- `IRenderSceneBridge::CreatePrimitive` 创建渲染侧代理
- 当变换变化时，它会标记渲染状态为 dirty
- `TWorld::SyncToScene()` 通过桥接层把 dirty 状态同步到渲染侧

这是当前项目最重要的架构接缝之一。

## 当前还没有的内容

当前渲染器尚未实现：
- 独立渲染线程
- 延迟渲染
- 可见性裁剪 Pass
- 阴影 Pass
- 材质图或完整材质系统
- 面向现代显式 API 的 Descriptor 驱动资源绑定模型

文档必须严格保持这一点，不能把规划内容写成现有能力。
