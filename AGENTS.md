# 项目级规则

## 文档新鲜度

本仓库将文档视为实现的一部分，而不是可选的后置工作。

规则：
- 任何涉及行为、架构、公共 API、模块布局、构建流程、运行流程、资源管线的修改，都必须在同一变更中同步更新 `Docs/` 下对应文档。
- 新增子系统、模块、可执行入口、测试目标或开发流程时，必须同步更新索引，确保 `Docs/文档总览.md` 始终是可靠的项目总入口。
- 如果发现现有文档与代码不一致，必须修正文档或删除过时内容，不能放任漂移存在。
- 优先描述当前已实现的代码。前瞻性设计必须明确标记为“计划中”或“待实现”，不能写成已经落地。
- 高层摘要放在索引文档中，深入细节放在 `Docs/architecture`、`Docs/guides`、`Docs/reference` 下。

代码变更时的必做流程：
1. 在结束任务前识别受影响的文档。
2. 在同一变更中更新这些文档。
3. 如果判断无需更新文档，也必须显式确认现有文档仍与代码一致。

## 架构与结构优化文档

- 当变更涉及程序结构调整、模块重组、依赖方向调整、生命周期管理调整、架构分层修改、渲染/资源/运行时主流程重构等“结构性或架构性修改”时，除更新常规文档外，还必须额外新增或更新一份独立的优化总结文档。
- 这份文档不能只混在原有说明文档里，必须作为独立文档存在，用于沉淀引擎开发过程中的设计经验。
- 文档内容至少必须说明：
  - 当前为什么要改，原有结构或架构存在什么问题
  - 这次修改的核心思路和设计取舍是什么
  - 修改后带来了哪些好处，解决了什么问题，仍然留下了什么限制
- 这类文档的目标读者是后续维护引擎的开发者，因此要偏“经验总结”和“设计复盘”，而不只是功能说明。
- 如果一次结构性修改拆成多个阶段进行，每个阶段都应在该文档中留下对应记录，或者新增分阶段文档并在索引中明确关联。
- 架构优化文档采用“增量记录”方式维护：新增阶段记录时不要覆盖或删除旧阶段内容，确保能够追溯历史路径与决策演进。
- 每次增量记录要控制篇幅，聚焦关键决策与关键影响，避免流水账式细节导致文档膨胀过快。
- 同时禁止过度简写：记录内容必须保证后续读者在脱离上下文时仍能理解“为何改、改了什么、效果如何”。

## 注释语言

- 代码注释尽量使用中文，保证项目内部说明风格一致。
- 变量名、类型名、接口名、协议名、图形 API 名称、专业术语等必须使用英文表达或保留英文时，对应注释不强制翻译成中文。

## AI 交互语言

- 在本仓库内与用户沟通时，默认使用中文回答。
- 如用户明确要求使用其他语言，或任务本身要求输出特定语言内容，可按该要求切换。

## AI 构建规则

- 在本仓库中使用 AI 或终端执行 CMake 构建时，默认使用 CLion 自带工具链，不要先切换到其他编译器再分析问题。
- 推荐使用以下 CMake 可执行文件：
  - `C:\Program Files\JetBrains\CLion 2025.3.3\bin\cmake\win\x64\bin\cmake.exe`
- 默认使用 `Release` 构建进行编译验证、性能排查和常规可用性确认；只有在明确需要调试符号、断点调试或排查 Debug 专属行为时，才切换到 `Debug`。
- 未特别说明时，优先使用 `cmake-build-release` 作为默认构建目录，不要默认回落到 `cmake-build-debug`。
- 如果使用 CLion 自带 MinGW 工具链，构建前必须确认以下目录已经加入当前进程的 `PATH`：
  - `C:\Program Files\JetBrains\CLion 2025.3.3\bin\mingw\bin`
- 如果直接调用 `cmake --build` 或 `g++.exe` 失败，必须先检查当前会话是否缺少上述 `PATH`，不要先假设源码有误。
- 在判断“项目无法编译”之前，必须先完成一次工具链环境自检；若补齐 CLion MinGW 的 `PATH` 后可以通过构建，应将问题归类为环境问题而不是代码问题。

## AI 架构修改原则

- 当“最小改动”与“更合理的长期架构”冲突时，默认优先选择长期合理架构，不以减少改动量为首要目标。
- 允许并鼓励必要的大规模重构（包括模块拆分、依赖重定向、生命周期重排、资源与渲染职责重构），前提是改动后架构边界更清晰、职责更稳定、演进阻力更低。
- 实施大改时必须同步提交对应文档与复盘记录，明确设计取舍、收益、限制与后续阶段计划。

本规则在本仓库中永久生效。
